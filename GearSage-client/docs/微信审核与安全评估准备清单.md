# 微信审核与安全评估准备清单

版本：v1.0  
状态：当前执行版  
适用对象：Tommy / Codex / 未来合规与审核协作

---

## 一、为什么要单独有这份清单

当前项目技术迁移已经基本完成，真正影响“能不能上线微信小程序”的核心，不再是服务器、数据库、上传，而是：

1. 个体户主体
2. ICP 备案
3. 论坛/社区类目的安全评估准备
4. 内容审核能力
5. 最小审核后台
6. 隐私政策 / 用户协议 / 注销说明
7. 小程序审核资料包

也就是说，当前最重要的不是继续堆功能，而是先把“微信审核所需的最小合规包”做出来。

---

## 二、当前结论

### 1. 论坛/社区类目不是绝对没路
但必须把以下事项当成硬门槛：

- 个体户主体
- ICP 备案
- 安全评估准备
- 内容审核
- 审核后台
- 用户协议 / 隐私政策 / 注销说明
- 举报与处置机制

### 2. 不能再把“能不能上线”理解成纯技术问题
现在真正决定是否能通过微信审核的，是：
- 主体与备案
- 制度与材料
- 审核能力
- 风险控制闭环

### 3. 当前最优策略
继续推进项目，但优先级切换为：

**先做合规前置包，再做产品深化。**

## 三、当前最高优先级（必须先做）

## P0：主体与备案前置
### 必做
- [ ] 个体工商户申请通过
- [ ] 小程序主体切换/确认使用个体户主体
- [ ] 域名与中国内地正式环境备案策略确认
- [ ] ICP 备案启动

### 说明
如果正式环境仍放在中国内地服务器，ICP备案是前置条件。  
这件事不能再后置。

---

## P1：微信审核最小合规包
### 必做
- [ ] 隐私政策
- [ ] 用户协议
- [ ] 账号注销说明
- [ ] 平台联系方式 / 投诉方式
- [ ] 举报入口设计
- [ ] 违规处理说明

### 说明
这些不是“附属资料”，而是审核材料的一部分。

---

## P2：安全评估准备包
### 必做
- [ ] 安全管理负责人明确
- [ ] 信息审核人员明确
- [ ] 审核制度文档
- [ ] 日志留存方案
- [ ] 投诉举报受理机制
- [ ] 技术配合机制
- [ ] 用户身份核验与留存策略说明
- [ ] 内容处置流程说明

### 说明
这一步不是先去“填表”，而是先把你自己内部制度和技术准备写出来。

---

## P3：审核能力上线

- [ ] 审核命中后的状态流转
- [ ] 驳回 / 下架 / 删除 / 恢复逻辑
- [ ] 人工复核入口

## 四、当前不应排在最前面的事项

以下事情仍然重要，但不应排在当前第一优先级：

- 商标证书结果
- Logo 终稿
- Android Studio / Xcode 深学
- App Store / Android 商店正式上架流程
- FinClip / Donut / Flutter / 原生的最终壳实现
- 大规模装备数据爬虫
- 大而全运营后台

原因：
这些事情都重要，但它们不会直接决定“微信审核这一步现在能不能走”。

---

## 五、当前推荐顺序（重新排序后）

### 第一层：先做
1. 个体户主体
2. ICP 备案
3. 隐私政策 / 用户协议 / 注销说明
4. 安全评估准备包
5. 内容审核接入

7. 正式短信接入

### 第二层：再做

9. 根据审核结果决定是否转 FinClip
10. Logo 终稿与品牌素材收口
11. 上架资料与多端打包流程

### 第三层：持续做
12. 装备数据生产机制
13. 产品深化（任务/商城/装备库）
14. 历史兼容逻辑清理

---

## 六、如果微信审核不通过

### 立即执行分支
- 不停止项目
- 不推翻独立后台
- 不重做业务逻辑
- 直接进入前端壳分支评估

### 当前建议顺序
1. FinClip
2. 再评估 Donut
3. 原生 / Flutter 暂不作为第一落点

---

## 七、你现在最该做的 5 件事

1. 等个体户审批，同时准备备案资料
2. 先把隐私政策 / 用户协议 / 注销说明补齐
3. 开始写安全评估准备包
4. 接入腾讯云内容安全
5. 做审核后台 v0

---

## 八、结论

当前阶段不是“继续把产品做厚优先”，而是：

> 先把微信审核与安全评估所需的最小合规包做出来，再决定微信这条路最终是否能走通。

这不会让项目变慢，反而能防止你继续在错误优先级上投入。
