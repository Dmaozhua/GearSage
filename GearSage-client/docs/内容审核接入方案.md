# 内容审核接入方案

版本：v2.0  
状态：P3-A 方案已拍板 / 待代码实施  
适用对象：Tommy / Codex / 后端 / 前端 / 未来审核协作

---

## 一、文档目标

这份文档用于定义 GearSage 在 P3-A 阶段的**腾讯云内容安全第一版接入落地方案**，并明确：

1. 第一版审核供应商与接入边界
2. 与当前 NestJS 项目结构一致的模块落点
3. 文本 / 图片审核分别接在哪些现有链路
4. 审核结果如何映射到当前业务状态
5. 审核后台 v0 如何消费审核记录

本文档是**实施方案**，不是泛泛建议。

---

## 二、当前拍板约束

本方案必须同时满足以下约束：

1. 与当前 `GearSage-api` 的 NestJS 结构一致
2. 不推翻当前返回结构 `{ code, message, data }`
3. 不改动当前已拍板上传口径：
   - `POST /upload`
   - `POST /upload/image`
   - `POST /upload/avatar`
   - `POST /upload/background`
4. 不回退到微信云开发能力
5. 不优先处理外网联调、备案、海外迁移

---

## 三、第一版结论

### 3.1 主供应商

第一版内容审核主供应商：**腾讯云内容安全**

原因：

1. 当前服务器、对象存储挂载、后续短信接入都在腾讯云体系
2. 文本与图片审核都能覆盖
3. 即使后续微信审核不通过转 FinClip / App，这层能力仍可复用

### 3.2 当前不采用

当前不作为第一版主线：

- 微信安全接口作为唯一审核主线
- 多供应商并行双审
- 自研审核模型
- 审核结果复杂评分引擎

第一版目标只有一个：

> 先形成“可解释、可留痕、可人工承接”的最小审核闭环。

---

## 四、与当前 NestJS 项目结构的对齐方式

## 4.1 当前项目真实结构

当前 `GearSage-api` 不是完整按 Nest feature module 拆到 `imports` 里，而是：

- `src/modules/*` 下按业务目录组织文件
- `src/app.module.ts` 直接注册 `controllers` / `providers`

因此第一版审核模块也应遵循这个现状，不额外引入重型模块化改造。

## 4.2 第一版建议新增目录

```text
src/modules/moderation/
  moderation.service.ts
  moderation.tencent.service.ts
  moderation.types.ts
  dto/
```

说明：

- 第一版**不要求**对小程序开放独立 `/moderation/*` 公共接口
- `moderation.service.ts` 主要给现有业务 service 内部调用
- `moderation.tencent.service.ts` 负责封装腾讯云 SDK / HTTP 调用
- `app.module.ts` 继续沿用当前方式，直接把新 provider 注册进去

## 4.3 不建议第一版新增的东西

第一版不建议做：

- 独立审核微服务
- 消息队列异步编排
- 多 provider 抽象工厂
- 独立 review worker 进程

这些都不是当前微信审核准备的阻塞项。

## 五、第一版审核范围

## 5.1 文本审核范围

第一版必审文本：

1. 用户昵称
2. 用户简介
3. 发帖标题
4. 发帖正文
5. 评论内容
6. 举报理由

第一版暂不优先：

1. 装备库官方参数
2. 系统内置文案
3. 后台管理员备注

## 5.2 图片审核范围

第一版必审图片：

1. 用户头像
2. 用户背景图
3. 发帖图片

第一版暂不优先：

1. 小程序静态资源
2. 装备库官方导入素材
3. 后台固定素材

---

## 六、接入策略拍板

## 6.1 文本审核策略

### 同步审核

适用：

1. 昵称
2. 用户简介
3. 评论
4. 举报理由

原因：

- 文本短
- 用户交互强
- 需要即时反馈

### 发帖发布前同步审核

适用：

1. 帖子标题
2. 帖子正文

第一版口径：

- 保存草稿时不审
- 点击发布时统一审核
- 命中 `PASS / REVIEW / REJECT`

## 6.2 图片审核策略

### 上传后同步审核

适用：

1. 头像
2. 背景图
3. 发帖图片

第一版口径：

- 继续走当前上传接口
- 后端收到文件后先写文件、登记 `media_assets`
- 然后立刻调用腾讯云图片审核
- 再决定是否允许这次上传成功返回

### 为什么第一版不做异步图片审核

因为当前有两个硬约束：

1. 不能改当前上传接口口径
2. 不希望前端先拿到一张“待审图片 URL”再补复杂状态机

所以第一版选择：

- `PASS`：直接按当前逻辑返回
- `REJECT`：直接阻断上传
- `REVIEW`：第一版按 `REJECT` 处理

这能最大限度保持当前上传链路不变。

---

## 七、审核结果模型

第一版统一抽象成 3 类结果：

### PASS

- 审核通过
- 允许进入正常业务流

### REJECT

- 明确违规
- 阻断当前提交
- 返回用户可读提示

### REVIEW

- 不确定 / 中风险
- 进入人工复核池

第一版注意：

- 对**上传链路**，`REVIEW` 暂时按 `REJECT` 处理
- 对**发帖链路**，`REVIEW` 进入待审核
- 对**评论链路**，`REVIEW` 允许入库但不展示
- 对**昵称 / 简介**，`REVIEW` 暂按 `REJECT` 处理

## 八、与现有业务表的最小落地方式

## 8.1 `bz_mini_topic`

当前已有：

- `status = 0` 草稿
- `status = 1` 待审核
- `status = 2` 已发布

第一版直接复用：

- 草稿保存：`0`
- 审核通过后发布：`2`
- 发布时命中 `REVIEW`：`1`
- 命中 `REJECT`：不进入正式发布

无需推翻现有帖子表。

## 8.2 `bz_topic_comment`

当前只有：

- `isVisible`

第一版建议最小加列：

- `status INT NOT NULL DEFAULT 2`

建议语义：

- `0` 待审核
- `2` 已展示
- `9` 已拒绝 / 已删除

同时保留 `isVisible`：

- `status = 2` 时 `isVisible = 1`
- `status = 0/9` 时 `isVisible = 0`

原因：

- 不推翻现有评论列表逻辑
- 但给后台和审核记录一个明确状态

## 8.3 `media_assets`

当前已有：

- `status`
- `bizType`
- `objectKey`
- `url`

第一版建议扩展列：

- `moderationStatus VARCHAR(16) NOT NULL DEFAULT 'pass'`
- `moderationProvider VARCHAR(32) NOT NULL DEFAULT ''`
- `moderationRiskLevel VARCHAR(32) NOT NULL DEFAULT ''`
- `moderationReason TEXT NOT NULL DEFAULT ''`
- `moderationRaw JSONB NOT NULL DEFAULT '{}'::jsonb`
- `moderatedAt TIMESTAMPTZ`

说明：

- 不改上传接口路径
- 不改 `data` 成功结构
- 只让资源表具备审核可追溯性

## 8.4 建议新增审核记录表

建议新增：

- `moderation_records`

最低字段：

- `id`
- `scene`
- `targetType`
- `targetId`
- `contentType`
- `provider`
- `result`
- `riskLevel`
- `riskReason`
- `rawResultJson`
- `operatorType`
- `operatorId`
- `createTime`

### 为什么必须有这张表

因为 P3-A 的目标不只是“能拦”，还要：

1. 能给微信说明你有审核能力
2. 能给审核后台展示命中记录
3. 能追溯一条内容为什么被挡

## 8.5 建议新增规则表

建议新增：

- `moderation_rules`

最低字段：

- `id`
- `ruleType`
- `matchType`
- `keyword`
- `status`
- `remark`
- `createTime`
- `updateTime`

第一版用途：

- 本地黑名单词
- 运营侧可解释的简单规则

## 九、业务链路接入点

## 9.1 用户资料更新

接入文件：

- `src/modules/user/user.service.ts`

第一版接入点：

1. 更新昵称前审核昵称
2. 更新简介前审核简介
3. 头像 / 背景图通过上传接口单独审核，不在 `user.service` 再重复审图

处理口径：

- `PASS`：正常更新
- `REJECT / REVIEW`：统一抛业务错误，资料不写库

## 9.2 评论发布

接入文件：

- `src/modules/comment/comment.service.ts`

第一版接入点：

1. `add()` 写库前先审评论文本

处理口径：

- `PASS`：按现有逻辑写评论，`status = 2`
- `REVIEW`：写评论但 `status = 0`、`isVisible = 0`
- `REJECT`：直接返回业务错误，不写库

兼容说明：

- 当前 `PUT /mini/comment` 的返回仍保持 `{ code, message, data: true }` 风格
- 前端当前依赖“发完后重拉评论列表”，因此待审核评论不会立即显示

## 9.3 帖子发布

接入文件：

- `src/modules/topic/topic.service.ts`

第一版接入点：

1. `publishTopic()` 内部审核标题
2. `publishTopic()` 内部审核正文
3. 草稿保存不审核

处理口径：

- `PASS`：`status = 2`
- `REVIEW`：`status = 1`
- `REJECT`：阻断发布，原草稿保留

兼容说明：

- 接口仍返回 `{ code, message, data }`
- `data.status` 将继续作为前端状态判断依据

## 9.4 上传图片

接入文件：

- `src/modules/upload/upload.service.ts`

第一版接入点：

1. 文件保存后
2. `media_assets` 写入后
3. 返回前同步调用图片审核

处理口径：

- `PASS`：维持当前成功返回 `{ fileName, objectKey, url }`
- `REJECT`：抛业务错误
- `REVIEW`：第一版按 `REJECT` 处理

注意：

- 不改路径
- 不改 multipart 上传方式
- 不要求前端改上传调用方式

## 9.5 举报理由

第一版落点：

- 若当前举报接口未实现，可先只在设计层预留
- 待举报功能落地时，统一走 `reviewText('report_reason', ...)`

## 十、模块与方法建议

## 10.1 `moderation.service.ts`

建议职责：

1. 统一定义场景
2. 统一先跑本地规则
3. 统一调腾讯云 provider
4. 统一记录 `moderation_records`
5. 输出标准化审核结果

建议方法：

- `reviewText(scene, content, metadata)`
- `reviewImage(scene, filePath, metadata)`
- `reviewTopicPublish(payload, metadata)`
- `reviewComment(content, metadata)`

## 10.2 `moderation.tencent.service.ts`

建议职责：

1. 封装腾讯云文本审核
2. 封装腾讯云图片审核
3. 统一把腾讯云原始响应映射成内部标准结果

内部标准返回建议：

```ts
type ModerationDecision = {
  result: 'PASS' | 'REJECT' | 'REVIEW';
  provider: 'tencent';
  riskLevel: string;
  riskReason: string;
  hitLabels: string[];
  raw: Record<string, any>;
};
```

## 10.3 本地黑名单词优先级

第一版建议：

1. 先跑本地黑名单词
2. 再调腾讯云内容安全

原因：

- 命中特别明确的词可直接阻断
- 也能给后台规则配置最小价值

---

## 十一、配置项建议

建议新增环境变量：

- `MODERATION_ENABLED=true`
- `MODERATION_PROVIDER=tencent`
- `TENCENT_MODERATION_SECRET_ID`
- `TENCENT_MODERATION_SECRET_KEY`
- `MODERATION_TEXT_REVIEW_ENABLED=true`
- `MODERATION_IMAGE_REVIEW_ENABLED=true`
- `MODERATION_REVIEW_AS_REJECT_ON_UPLOAD=true`

说明：

- 第一版以开关控制为主
- 本地仍可保留关闭审核的联调能力

---

## 十二、接口返回与前端文案口径

## 12.1 统一错误文案

不要把供应商原始错误直接透给用户。

第一版统一为：

- 文本违规：`内容不符合社区规范，请修改后重试`
- 图片违规：`图片内容不符合社区规范，请更换后重试`
- 帖子待审核：`内容已提交，审核通过后展示`

## 12.2 不改变当前返回结构

第一版明确不推翻：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

允许的变化：

- `data.status` 在发帖链路中出现 `1`
- 上传 / 评论成功时内部逻辑变化，但对小程序成功结构保持兼容

---

## 十三、第一版实施顺序

建议顺序：

1. 新增 `moderation_records`、`moderation_rules` 与必要字段
2. 新增 `src/modules/moderation/*`
3. 先接文本审核：
   - 用户昵称 / 简介
   - 评论
   - 发帖标题 / 正文
4. 再接图片审核：
   - 头像
   - 背景图
   - 发帖图
5. 最后打通审核后台 v0 的待审核与记录查询

---

## 十四、当前一句话结论

腾讯云内容安全第一版不是单纯“加一个 SDK”，而是：

> **在不推翻现有接口与上传口径的前提下，把审核能力嵌进 `user / topic / comment / upload` 四条现有主链，并留下后台可承接的审核记录。**
