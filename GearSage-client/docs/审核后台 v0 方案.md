# 审核后台 v0 方案

版本：v1.0  
状态：P3-A 当前执行版  
适用对象：Tommy / Codex / 后端 / 前端 / 未来运营

---

## 一、文档目标

这份文档用于定义 GearSage 的审核后台 v0。

当前目标不是做大而全运营平台，而是做一个：

> **在微信审核与日常风控场景下，够用、能演示、能处理问题的最小后台。**

---

## 二、为什么必须先做 v0

你现在需要的不只是“有审核接口”，而是：

1. 有地方看待审核内容
2. 有地方处理待审核帖子和评论
3. 有地方封禁或解封用户
4. 有日志能证明“谁处理了什么”

如果没有后台，即使接了审核 API，也很难向平台或自己证明你有治理能力。

---

## 三、v0 目标

审核后台 v0 的目标只有 4 个：

1. 管理员能登录
2. 能看待审核内容
3. 能处理帖子 / 评论 / 用户
4. 能留下审核痕迹

---

## 四、v0 范围（只做这些）

### 4.1 管理员登录
最低需求：
- 管理员账号密码登录
- 先只支持 1 个超级管理员

### 4.2 帖子审核
最低需求：
- 查看待审核帖子
- 查看帖子详情
- 通过
- 驳回
- 删除/下架

### 4.3 评论审核
最低需求：
- 查看待审核评论
- 删除
- 恢复（可选）
- 标记处理完成

### 4.4 用户处理
最低需求：
- 查看用户基本信息
- 封禁
- 解封

### 4.5 审核日志
最低需求：
- 记录管理员
- 记录动作
- 记录时间
- 记录备注

### 4.6 简单规则配置
最低需求：
- 黑名单词
- 审核备注模板（可选）

---

## 五、v0 明确不做

以下不作为第一版范围：

- 复杂 RBAC 权限系统
- 多角色审批流
- 统计大盘
- 工单流
- 运营活动配置中心
- 高级搜索 BI
- 审核分单系统
- 批量风控模型

原因：
这些都不会决定你现在能不能先过微信审核。

最低字段：
- `id`
- `username`
- `passwordHash`
- `status`
- `role`
- `createTime`
- `updateTime`

## 7.2 建议新增后台操作日志表
建议表：
- `admin_operation_logs`

最低字段：
- `id`
- `adminUserId`
- `targetType`
- `targetId`
- `action`
- `remark`
- `createTime`

---

## 7.3 建议新增规则表
建议表：
- `moderation_rules`

最低字段：
- `id`
- `type`
- `keyword`
- `status`
- `createTime`
- `updateTime`

- `bz_mini_topic`
- `bz_topic_comment`
- `bz_mini_user`

需要做的是：
- 增加审核状态字段或明确状态语义
- 能让后台列表筛待审核 / 已驳回 / 已封禁

## 八、接口建议

## 8.1 后台鉴权
建议前缀：
- `/admin/auth/*`

最少接口：
- `POST /admin/auth/login`
- `POST /admin/auth/logout`
- `GET /admin/auth/me`

---

## 8.2 帖子审核
建议接口：
- `GET /admin/topic/review/list`
- `GET /admin/topic/review/detail?id=:id`
- `POST /admin/topic/review/pass`
- `POST /admin/topic/review/reject`
- `POST /admin/topic/review/remove`

---

## 8.3 评论管理
建议接口：
- `GET /admin/comment/list`
- `POST /admin/comment/remove`
- `POST /admin/comment/restore`

---

## 8.4 用户管理
建议接口：
- `GET /admin/user/list`
- `POST /admin/user/ban`
- `POST /admin/user/unban`

---

## 8.5 审核日志
建议接口：
- `GET /admin/logs`

---

## 8.6 规则配置
建议接口：
- `GET /admin/moderation/rules`
- `POST /admin/moderation/rules`
- `DELETE /admin/moderation/rules?id=:id`

---

## 九、技术实现建议

## 9.1 前端形态
第一版建议：
- 不要先做完整独立管理端工程
- 先做一个最简单的后台前端即可

你有三个可选形态：

### 方案 A：Nest + 模板页
最小、快，但后续扩展差

### 方案 B：单独一个极简 React / Vue 管理端
我更建议这个
- 可快速做表格和表单
- 后续扩展比模板页更顺手

### 方案 C：直接买/套大后台框架
当前不建议
- 容易把你带偏到做后台本身

### 当前建议
**方案 B：极简 React 管理端**

---

## 9.2 与业务系统关系
审核后台不是独立业务系统。  
它应直接操作当前独立后台里的：

- topic
- comment
- user
- moderation

这样后续最稳。

---

## 十、第一版实施顺序

1. 管理员登录
2. 帖子审核列表 + 通过/驳回
3. 评论列表 + 删除
4. 用户封禁/解封
5. 审核日志
6. 黑名单词

---

## 十一、给微信审核看的重点能力

如果后续要向平台解释，你最应该能展示的是：

1. 你有待审核内容列表
2. 你有人工审核入口
3. 你能封禁违规用户
4. 你有日志留痕
5. 你有黑名单词
6. 你不是放任用户随便发内容

所以 v0 已经足够支撑“治理能力展示”。

---

## 十二、Codex 当前最适合接手的部分

1. `admin` 模块后端接口骨架
2. 管理员表与日志表 SQL
3. 帖子审核状态流转接口
4. 评论管理接口
5. 最简单后台前端骨架页

---

## 十三、当前一句话结论

审核后台 v0 的目标不是漂亮，而是：

> **让 GearSage 具备最小可演示、可处理、可留痕的内容治理后台。**
