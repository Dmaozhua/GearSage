# 审核后台 v0 方案

版本：v2.0  
状态：P3-A 技术方案已拍板 / 待代码实施  
适用对象：Tommy / Codex / 后端 / 前端 / 未来运营

---

## 一、文档目标

这份文档用于定义 GearSage **审核后台 v0 的技术落地方案**。

当前目标不是做一个完整运营平台，而是做一个：

> **能支撑微信审核说明、能处理待审核内容、能留痕的最小后台。**

---

## 二、当前拍板约束

本方案必须同时满足：

1. 后端与当前 NestJS 项目结构一致
2. 继续沿用 `{ code, message, data }` 返回结构
3. 不推翻现有小程序主链接口
4. 不回退微信云开发
5. 优先做“可审核、可说明、可演示”，不是大后台

---

## 三、第一版落地结论

### 3.1 后端落点

审核后台 v0 后端继续放在：

- `GearSage-api`

不新开独立 admin backend 仓库。

原因：

1. 当前所有业务数据都在这一套 PostgreSQL
2. 审核后台本质上是当前业务系统的治理侧
3. 新开独立后端会增加鉴权、部署、联调成本

### 3.2 前端落点

第一版建议：

- **不优先新开大而全管理端仓库**
- 先做一个最小后台页面层

推荐方案：

1. 后端 API 在 `GearSage-api`
2. 后台页面第一版可以是：
   - 极简静态页面
   - 或极简 React 构建产物
3. 由 Nest / Nginx 统一托管

当前不建议：

- 先搭完整中后台框架
- 先做复杂权限系统
- 先做图表大盘

---

## 四、与当前 NestJS 项目结构的对齐方式

## 4.1 当前真实结构

当前项目以：

- `src/modules/*` 目录组织代码
- `src/app.module.ts` 直接注册 controllers / providers

为主。

因此审核后台 v0 建议新增：

```text
src/modules/admin/
  admin-auth.controller.ts
  admin-auth.service.ts
  admin-review.controller.ts
  admin-review.service.ts
  admin-user.controller.ts
  admin-user.service.ts
  admin-log.controller.ts
  admin-log.service.ts
  admin-rule.controller.ts
  admin-rule.service.ts
  dto/
```

并在：

- `src/app.module.ts`

中继续直接注册 controllers / providers。

### 4.2 为什么不先引入复杂 admin module 体系

因为当前阶段重点是：

- 审核能力闭环
- 微信审核可说明
- 最小演示后台

而不是做中后台基础设施工程。

---

## 五、v0 的范围重新拍板

审核后台 v0 只做以下 5 类能力：

1. 管理员登录
2. 帖子审核
3. 评论审核
4. 用户封禁 / 解封
5. 审核日志与简单规则

第一版明确不做：

- RBAC 权限系统
- 多级审批流
- 仪表盘 BI
- 批量风控工单
- 多角色后台
- 运营活动配置中心

## 六、数据层设计

## 6.1 建议新增表

### `admin_users`

最低字段：

- `id`
- `username`
- `passwordHash`
- `status`
- `role`
- `lastLoginAt`
- `createTime`
- `updateTime`

- 不复用 `bz_mini_user.isAdmin` 做后台登录主体

### `admin_operation_logs`

最低字段：

- `id`
- `adminUserId`
- `targetType`
- `targetId`
- `action`
- `remark`
- `extra`
- `createTime`

说明：

- 用于记录后台通过 / 驳回 / 删除 / 封禁等动作

### `moderation_rules`

最低字段：

- `id`
- `ruleType`
- `matchType`
- `keyword`
- `status`
- `remark`
- `createTime`
- `updateTime`

说明：

- 第一版只做关键词黑名单即可

## 6.2 复用现有业务表

继续复用：

- `bz_mini_topic`
- `bz_topic_comment`
- `bz_mini_user`
- `media_assets`
- `moderation_records`

其中建议最小补列：

### `bz_topic_comment`

- `status`

### `media_assets`

- `moderationStatus`
- `moderationReason`
- `moderationProvider`

### `bz_mini_user`

当前 `status` 字段继续用于用户封禁语义，但需要在文档中明确：

- `0` 正常

说明：

- 第一版不要求重做用户状态体系
- 先采用最小明确化

## 七、后台接口落地方案

所有后台接口继续保持：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

## 7.1 管理员鉴权

前缀：

- `/admin/auth/*`

接口：

- `POST /admin/auth/login`
- `POST /admin/auth/logout`
- `GET /admin/auth/me`

第一版建议：

- 后台账号密码登录
- 独立 `ADMIN_JWT_SECRET`
- 不复用小程序 access token
- 第一版可以不做 refresh token

原因：

- 管理后台用户极少
- 简化最快

## 7.2 审核列表与审核动作

建议统一做成 review controller：

- `GET /admin/review/topics`
- `GET /admin/review/topics/:id`
- `POST /admin/review/topics/:id/pass`
- `POST /admin/review/topics/:id/reject`
- `POST /admin/review/topics/:id/remove`

- `GET /admin/review/comments`
- `GET /admin/review/comments/:id`
- `POST /admin/review/comments/:id/pass`
- `POST /admin/review/comments/:id/reject`
- `POST /admin/review/comments/:id/remove`

### 为什么评论也要 pass / reject

因为 P3-A 内容审核第一版会引入：

- 评论 `PASS`
- 评论 `REVIEW`
- 评论 `REJECT`

如果后台没有 `pass / reject`，就无法承接人工复核。

## 7.3 用户管理

建议接口：

- `GET /admin/users`
- `GET /admin/users/:id`
- `POST /admin/users/:id/ban`
- `POST /admin/users/:id/unban`

第一版足够。

## 7.4 日志与规则

建议接口：

- `GET /admin/logs`
- `GET /admin/rules`
- `POST /admin/rules`
- `DELETE /admin/rules/:id`

第一版黑名单词即可，不做复杂规则 DSL。

## 九、与内容审核接入的衔接方式

## 9.1 帖子

后台主要消费：

- `bz_mini_topic.status`
- `moderation_records`

动作映射：

- `pass` -> `status = 2`
- `reject` -> `status = 1` 保留待处理痕迹或补拒绝标记
- `remove` -> `isDelete = 1`

第一版建议：

- 真正“驳回原因”不要只塞在帖子 `extra`
- 应至少写入 `admin_operation_logs`

## 9.2 评论

后台主要消费：

- `bz_topic_comment.status`
- `bz_topic_comment.isVisible`
- `moderation_records`

动作映射：

- `pass` -> `status = 2`, `isVisible = 1`
- `reject` -> `status = 9`, `isVisible = 0`
- `remove` -> `status = 9`, `isVisible = 0`

## 9.3 用户

后台主要消费：

- `bz_mini_user.status`

动作映射：

- `ban` -> `status = 9`
- `unban` -> `status = 0`

## 十、守卫与安全设计

第一版建议新增：

- `src/common/admin-jwt-auth.guard.ts`
- `src/common/current-admin.decorator.ts`

不要复用小程序用户 guard。

原因：

1. 管理员 JWT secret 独立
2. payload 不同
3. 后续日志与权限扩展更清楚

---

## 十一、第一版实施顺序

建议按以下顺序实施：

1. 建 `admin_users`、`admin_operation_logs`、`moderation_rules`
2. 做后台管理员登录
3. 做待审核帖子列表与通过 / 驳回
4. 做评论审核列表与通过 / 驳回 / 删除
5. 做用户封禁 / 解封
6. 做审核日志
7. 最后补最小规则页

---

## 十二、当前一句话结论

审核后台 v0 的技术方案不是做一个新平台，而是：

> **在当前 `GearSage-api` 里新增 `admin` 治理侧接口与最小后台页面，用最小表结构和最小动作集合承接内容审核记录、人工复核和用户处置。**
