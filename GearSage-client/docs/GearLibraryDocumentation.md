# 微信小程序装备库逻辑说明文档

## 1. 概述

当前小程序的装备库模块是一个功能完善的数据展示与查询系统。它通过 **Excel 管理数据源**，使用 **Node.js 脚本**进行数据导入和处理，最终在**小程序端**为用户提供分类浏览、列表筛选、关键词搜索和详情对比等功能。

整个系统的核心设计思想是**数据驱动**和**前后端分离**，并通过多种策略优化用户体验，尤其是在数据查询和筛选方面。

## 2. 数据采集前置层 (v2.0+)

随着系统发展，原有的手动维护Excel模式已升级为一套半自动化的数据采集前置流程，其核心是 **Scrapling MCP (Micro-Crawling Platform)**。这一层确保了数据源的权威性、准确性和采集过程的可复现性。

### 2.1. 核心组件

*   **Scrapling MCP 服务**: 一个在本地运行的、负责执行所有采集任务的中央服务。它管理并发、处理反爬策略并提供统一的API。
*   **`source_config.json`**: 定义了所有目标品牌和品类的权威数据源入口URL。
*   **采集器配置 (YAML)**: 位于 `scripts/scrapers/` 目录下，为每个采集任务详细定义了抓取规则、CSS选择器和字段映射逻辑。

### 2.2. 新的数据流

新的数据采集流程在原有的“数据导入”之前增加了一个“采集与标准化”阶段：

1.  **采集 (Scrapling MCP)**: 根据采集器配置，从权威官网抓取原始HTML数据。
2.  **标准化**: MCP服务或其调用的AI模块，将非结构化的HTML解析为统一的 **`normalized.json`** 格式。这份中间数据是后续所有处理的基石。
3.  **校验**: `pre-check.js` 脚本对 `normalized.json` 进行严格的Schema校验和唯一性检查。
4.  **转换**: 校验通过后，专用的转换脚本将 `normalized.json` 转换为符合下游 `import_gear_excel.js` 脚本要求的多个Excel文件。

这个前置层将数据生产的起点从“手动填写Excel”前移到了“从官网自动采集”，极大地提升了数据录入的效率和准确性。

### 2.3. 自动化分类与多语言处理 (v2.2+)

在标准化的基础上，系统引入了更高级的数据预处理逻辑：

1.  **动态枚举推导 (`classifyLure` 等)**: 脚本不再仅仅是原样搬运数据，而是能够根据装备的名称、规格（如深度提取、浮力、包含数量）以及官网大分类，自动推导出符合 GearSage 系统的标准枚举值（如 `type_tips`, `system`, `water_column`, `action`）。
2.  **多语言描述回退机制**: 针对具有多语言支持的官网（如 Megabass），爬虫实现了优先级回退抓取（优先有效中文 > 英文 > 日文），并在抓取过程中自动清洗无用节点（如 `script`, `style`, 语言切换器），确保描述文本的高质量与纯净。
3.  **动态多 Sheet 导出**: 在将 `normalized.json` 转换为 Excel 时，转换脚本（如 `to_excel_megabass_lure.js`）会根据推导出的 `system` 字段（如 hardbait, soft, wire）自动将数据路由并拆分到不同的 Excel Sheet 中，直接生成符合最终导入格式的文件，统一输出到 `data_raw/` 目录下。
4.  **共享导出规范**: `scripts/gear_export_schema.js` 维护品牌 ID、sheet 名和标准 header。`data_raw` 导出层默认直接对齐 `rate/excel` 的最终字段和主键风格，尽量减少人工二次统一格式。
5.  **差异报告**: `scripts/report_rate_excel_diffs.js` 会对比 `pkgGear/data_raw/*_import.xlsx` 与 `rate/excel` 当前基准切片，输出只读 markdown 报告到 `pkgGear/data_raw/rate_excel_diff_report.md`，用于人工确认是否需要回填最终表。
6.  **最终基准约定**: `rate/excel` 是最终使用数据，`pkgGear/data_raw` 是抓取/清洗/导出中间层。扩充时默认先跑差异报告，不直接自动覆盖最终总表。日常执行可直接参考 [装备扩充操作清单_v1.md](/Users/tommy/GearSage/GearSage-client/docs/装备扩充操作清单_v1.md)。
7.  **导入前校验**: `GearSage-api/scripts/import_gear_excel.js` 现已支持 `--dry-run`。推荐在正式导库前先执行 `npm run import:gear:check`，只校验 `rate/excel` 的品牌引用、主键、外键与重复 key，不连接数据库、不执行清表。

## 3. 数据流与核心流程

装备库的数据流向清晰，可以分为三个主要阶段：

1.  **数据源 (Excel):**
    *   所有装备数据（包括品牌、主型号、详细规格）均在 `GearSage-client/rate/excel/` 目录下的多个 Excel 文件中进行维护。
    *   这种方式使得非技术人员也能方便地管理和更新装备数据。
    *   *注：自 v2.3 起，`rate/excel` 被明确固定为最终基准。`pkgGear/data_raw` 的自动转换产物需要先经过差异报告核对，再决定是否回填最终总表。*

2.  **数据导入与处理 (Node.js):**
    *   位于 `GearSage-api/scripts/import_gear_excel.js` 的脚本是连接数据源和后端服务的桥梁。
    *   **执行时机:** 当装备数据需要更新时，手动执行此脚本。
    *   **核心任务:**
        1.  **读取 Excel:** 解析所有相关的 `.xlsx` 文件。
        2.  **数据清洗:** 对数据进行格式化、规范化处理。
        3.  **导入数据库:** 将清洗后的数据存入 PostgreSQL 数据库的三张核心表中 (`gear_brands`, `gear_master`, `gear_variants`)。此过程是**全量覆盖**，即先清空表再插入新数据。
        4.  **生成客户端搜索文件:** 从主数据中提取关键信息（型号、年份、别名等），生成一个 `Data.js` 文件，并放置在小程序项目目录 `GearSage-client/pkgGear/searchData/` 下。

3.  **数据消费 (小程序端):**
    *   小程序通过 API 接口从后端获取数据进行展示，并通过 `Data.js` 文件增强本地搜索能力。
    *   用户在小程序中的操作流程通常是：
        *   **首页 (`index`):** 选择一个大的装备分类（如渔轮、鱼竿、鱼饵、鱼线）。
        *   **列表页 (`list`):** 查看该分类下的装备列表，可以进行**分页浏览**、**后端筛选**或**前端关键词搜索**。
        *   **详情页 (`detail`):** 点击列表项进入，查看该型号下所有不同规格（SKU）的详细参数对比表格。

## 3. 数据结构

### 3.1. Excel 表格格式与结构差异

数据被组织在多个结构化的 Excel 文件中。不同类型的装备，其详细规格文件（`*_detail.xlsx`）的列结构**完全不同**，这体现了各类装备属性的差异性。

*   **`brand.xlsx` (品牌):**
    *   `id`, `name`, `name_en`, `name_jp`, `name_zh`, `description`, `site_url`

*   **`reel.xlsx`, `rod.xlsx`, `lure.xlsx` (主数据):**
    *   这些文件共享一套通用字段结构。
    *   `id`: 装备唯一标识。当前约定为**字符串前缀主键**（如 `SL1000`, `DL1000`, `MR1000`, `DLN1000`）。
    *   `brand_id`: 关联品牌，保持为**纯数字品牌 ID**。
    *   `model`, `model_cn`, `model_year`: 型号、中文名、年份。
    *   `system`, `water_column`, `action`: **核心筛选维度**，定义了装备的基础分类属性（如：硬饵、水面、走狗）。这是实现路亚饵智能筛选的基础。
    *   `alias`, `type_tips`: 用于增强搜索和展示的辅助信息。
    *   `images`: 图片。

*   **`*_detail.xlsx` (变体/规格数据):**
    *   **`spinning_reel_detail.xlsx` (纺车轮):** 包含 `SKU`, `GEAR RATIO` (速比), `DRAG` (卸力), `WEIGHT` (自重) 等渔轮专属规格。
    *   **`rod_detail.xlsx` (鱼竿):** 包含 `SKU`, `TOTAL LENGTH` (全长), `Action` (调性), `PIECES` (节数), `LURE WEIGHT` (饵重范围) 等鱼竿专属规格。
    *   **`hardbait_lure_detail.xlsx` (硬饵):** 包含 `SKU`, `Length` (长度), `Weight` (重量), `Buoyancy` (浮力状态) 等路亚饵专属规格。
    *   这种结构差异是设计上的核心，它允许每种装备类型都有自己独特的属性集。

### 3.2. 数据库表结构

导入脚本将 Excel 数据存入以下三张 PostgreSQL 表：

*   `gear_brands`: 存储品牌信息。
*   `gear_master`: 存储装备的主信息，与主数据 Excel 文件对应。
*   `gear_variants`: 存储装备的详细规格（变体），通过 `gearId` 与 `gear_master` 关联。所有非标准化的、来自不同 `*_detail.xlsx` 文件的规格参数都存储在 `raw_json` 字段中，以应对结构差异。

### 3.3. 客户端搜索数据 (`Data.js`)

这是一个由导入脚本生成的 JavaScript 文件，其内容是一个包含所有装备核心信息的数组。

```javascript
// GearSage-client/pkgGear/searchData/Data.js
module.exports = [
  {
    "type": "reel",
    "id": "SR1000",
    "name": "2022 STELLA C3000SDH",
    "alias": "斯特拉",
    "type_tips": "双摇把"
  },
  // ...更多装备
];
```

*   **作用:** 使小程序可以在本地直接对整个装备库进行快速、模糊的文本搜索，而无需每次都请求后端，极大地提升了搜索体验。

## 4. 小程序端实现逻辑

### 4.1. 页面结构

*   `pkgGear/pages/index`: **分类入口页**。UI上提供如“渔轮”、“鱼竿”、“路亚饵”、“鱼线”等入口，点击后将分类 `type` 传递给列表页。
*   `pkgGear/pages/list`: **列表与查询核心页**。
*   `pkgGear/pages/detail`: **详情展示页**。
*   `pkgGear/pages/compare`: **同类子型号对比页**。当前第一版仅对渔轮、鱼竿开放；鱼饵和鱼线先以列表筛选、详情阅读、关联内容承接为主，消费本地对比池与 `GET /mini/gear/detail` 返回的 `variants` 数据。

### 4.2. 列表页 (`list`) 的查询与筛选机制

列表页是整个装备库最核心、逻辑最复杂的部分。它巧妙地结合了多种数据查询策略：

*   **默认浏览:**
    *   **后端分页** (`apiService.getGearList` with `page` and `pageSize`)。
    *   用户滚动到底部时加载下一页。
    *   **优点:** 首次加载快，节省流量。

*   **筛选 (Filter):**
    *   筛选功能是**后端筛选**和**前端筛选**的混合体，针对不同装备类型采用不同策略。
    *   **通用后端筛选:** 对于渔轮、鱼竿等筛选维度明确的品类，筛选参数直接传递给 `getGearList` 接口，由数据库完成过滤。
    *   **路亚饵的本地二次筛选:**
        *   **场景:** 当用户在路亚饵分类下，没有输入关键词，而是通过筛选面板选择了“推荐词”（如“胖子”、“VIB”等）。
        *   **流程:** 小程序首先从后端获取一个经过基础筛选的列表，然后 `onFilter` 事件会触发 `filterGearListByRecommendations` 函数。
        *   **目的:** 这个函数在**小程序本地**执行 `筛选设计.txt` 中定义的**推导规则**，例如将 `硬饵 + 中层 + 摇摆` 的装备识别为“胖子”，从而实现更智能的分类筛选。这是路亚饵筛选逻辑的核心。

*   **搜索 (Search):**
    *   用户输入关键词后，触发 `onSearch`。
    *   **前端搜索:**
        1.  向后端请求一个包含**几乎所有数据**的大列表 (`pageSize: 500`)。
        2.  在小程序端，使用 `filterGearListByKeyword` 函数，将用户输入的关键词与 `Data.js` 中为每个装备注入的 `name`, `alias`, `type_tips` 等信息进行匹配。
    *   **优点:** 搜索响应极快，支持对别名、俗称等非标准名称的模糊匹配，体验好。
    *   **权衡:** 需要一次性加载较多数据，在数据量极大时可能会有性能瓶颈。

### 4.3. 详情页 (`detail`) 的分层展示

详情页当前已从“单张动态大表”升级为分层阅读结构：

*   顶部识别区：品牌、主型号、图片、基础标签、加入对比、去求推荐。
*   子型号选择区：优先让用户先选 `variant`，再查看核心参数与完整规格。
*   核心参数区：按装备类型提取最常被用于排除候选的 5~8 个字段。
*   完整规格区：继续基于 `variants` 动态字段生成，但按阅读顺序做分组展示。
*   GearSage 提示区：基于当前已收录子型号差异做轻量解释，不冒充厂家官方参数。
*   系列子型号速览：保留横向规格表，用于快速扫一眼同系列差异。

### 4.4. 对比页 (`compare`) 的第一版逻辑

对比页当前不新增后端接口，而是复用 `GET /mini/gear/detail`：

*   详情页先把用户选中的 `variant` 放入本地对比池。
*   对比页读取 2~3 个候选，按 `masterId + variantKey` 回查详情数据。
*   第一版限制为：
    1.  同类型比较
    2.  同子类方向比较
    3.  最多 3 个候选
*   默认展示：
    *   L1 核心参数差异
    *   完整规格对比（默认只看有差异的字段）
    *   GearSage 提示
    *   去求推荐承接

### 4.5. 当前增量返回字段口径

在不改动原有主链路字段的前提下，gear 返回当前新增三层增量结构：

*   `official_specs`
    *   从当前 master / variant 的官方字段中整理出的可直接比较参数
    *   目的是减少前端继续直接遍历 `raw_json`
*   `gsc_traits`
    *   GearSage 解释层字段
    *   当前允许为空
    *   当前若 Excel 未显式维护，会由后端按现有字段做轻量归纳
*   `compare_profile`
    *   对比页专用聚合信息
    *   当前包含同类比较所需的 `compareGroup / coreFieldKeys / warningHints / fitStyleTags` 等轻量字段

当前原则：

*   不删除原有平铺字段
*   不改变 `GET /mini/gear/list` / `GET /mini/gear/detail` 的主 envelope
*   前端优先消费新结构，缺失时再回退旧字段

## 5. 总结与建议

该装备库系统架构设计良好，在数据维护的便利性、查询性能和用户体验之间取得了很好的平衡。

*   **优点:**
    *   数据维护入口统一（Excel），方便易用。
    *   通过生成 `Data.js` 实现了高性能的客户端搜索。
    *   针对不同装备类型设计了不同的筛选策略，兼顾了精确性（后端筛选）和灵活性（路亚饵的本地推导筛选）。
    *   详情页动态表格能完美适应不同品类装备的规格展示。

*   **潜在风险/可优化点:**
    *   当前端搜索依赖一次性加载大量数据。如果未来装备数量激增（例如超过 1000-2000 条），可能会导致小程序在搜索时加载时间变长或内存占用过高。届时可考虑将搜索功能也移至后端实现，例如使用 Elasticsearch 或数据库的全文检索功能。
    *   数据导入脚本是手动执行的，存在忘记执行或执行出错的风险。未来可考虑将其集成到 CI/CD 流程中，或提供一个简单的后台管理界面来触发更新。
