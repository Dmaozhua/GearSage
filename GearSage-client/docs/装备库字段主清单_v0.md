## 一、文档定位

这份文档不是字段解释手册，也不是单纯的字段分层规范。

它的作用是把：

- 《装备库字段解释手册 v1》里的**当前已有字段**
- 《装备库字段扩充与爬取分层规范 v1》里的**字段分层与扩充口径**

合并成一份可执行的**字段施工总表**。

也就是：

1. 现在具体有哪些字段值得做
2. 它属于哪一层
3. 它主要服务 detail、compare 还是 both
4. 当前是已有、缺失还是定义模糊
5. 是否值得进入第一版主对比
6. 后续应该优先走哪条落地路径

## 三、统一标注口径

### 3.1 字段层级

- `official_specs`：官方参数 / 官方结构化说明 / 可稳定采集
- `gsc_traits`：GearSage 归一、解释、提示、归纳
- `deep_fields`：深玩家字段 / 改装字段 / 实测字段 / 覆盖不稳定但高价值
- `supplement`：补充说明、品牌技术、来源信息、证据链、营销信息

### 3.2 场景范围

- `detail`：更适合单品深读
- `compare`：更适合横向对比
- `both`：两边都重要

### 3.3 展示位置

- `list`：列表页可展示
- `detail_core`：详情核心参数区
- `detail_full`：详情完整规格区
- `compare_core`：对比页主对比
- `deep_read`：深读区 / 改装区 / 补充区
- `hidden_runtime`：运行时使用，前端不必直接展示

### 3.4 当前状态

- `已有`：当前 Excel / 导入链已有较明确承载
- `缺失`：当前没有稳定字段承载
- `模糊`：有类似字段，但口径不稳、覆盖不稳或语义未定

### 3.5 优先级

- `P0`：现在就值得推进
- `P1`：重要但可后置
- `P2`：先预留，不抢当前主线

### 3.6 落地路径

- `excel`：建议进最终表结构 / 最终表字段
- `derive`：运行时归一 / 后端推导
- `raw_json`：先保留在原始明细或补充 JSON
- `manual`：人工维护
- `player`：玩家实测 / 后续补录

### 3.7 主表通用补充字段

- `Description`
  - 含义：当前主商品在官网的描述文字
  - 层级：`supplement`
  - 主要场景：`detail`
  - 建议展示位置：`deep_read`
  - 口径说明：这是**主商品 / 系列层**的官网描述，不等于子型号描述，也不应冒充 GearSage 总结
  - 当前建议：保留在主表，用于详情页深读区，不进入第一版主对比

## 四、执行原则

1. 先补真正影响 detail / compare 的字段，不追求一次性大全。  
2. 玩家真正关心但官网通常不给的字段，必须承认其属于 `deep_fields`。  
3. 第一版主对比只吃稳定字段，不把覆盖低的改装字段硬塞进去。  
4. 默认不轻易动 `rate/excel` 总表结构；能先通过导入归一、后端归一或 `raw_json` 承接的，先不要大改表。  
5. 未确认字段宁可暂不展示，也不要冒充确定事实。  
6. 品牌技术、官方宣传点只进补充层，不进主对比。

#### 5.5.1 建议进 Excel 的字段

#### 5.5.1.a reel 填写规则补充

- `is_sw_edition`
  - 统一填写：`1` / `0`
  - 不再使用 `是/否`
- `Description`
  - 主表填写主商品 / 系列层官网描述
  - detail 若需要规格级描述，写在 `variant_description`
- `official_environment`
  - 优先保留官网明确环境口径，如 `freshwater` / `saltwater`
  - 没有明确口径时可留空，不要猜
- `usage_environment`
  - 可以填写更贴近使用场景的说明，如 `近岸` / `船钓` / `远海`
  - 若和 `official_environment` 冲突，以官网明确口径优先
- `line_capacity_display`
  - 面向用户展示的一条主容线表达
  - 尽量写成整理后的完整可读文本，不要只写半截
- `spool_weight_g / spool_axis_type / knob_size`
  - 有可靠来源再填，没有资料留空
  - 不允许凭经验猜值
- `battery_capacity / battery_charge_time / continuous_cast_count`
  - 仅适用于特殊/电动轮型
  - 传统机械轮默认留空

## 六、鱼竿字段主清单

### 6.1 设计目标

鱼竿这类的核心难点，不在于“参数少”，而在于：

- 纸面参数和真实手感之间往往有落差
- 同样是长度 / 调性 / 硬度，落到不同场景上意义完全不同
- 很多玩家真正在乎的是“这根竿到底偏什么、甜区在哪、累不累、适不适合我现在这套玩法”

所以鱼竿字段设计必须同时服务两个层：

1. **决策主层**

2. **深读层**  
   再让用户理解：这根竿的定位、平衡感、甜区、腰力、导环和接节体验这些深层信息。

## 七、鱼饵字段主清单

### 7.1 设计目标

鱼饵这类的核心难点，不在于“参数少”，而在于：

- 纸面上的颜色、尺寸、重量，只能回答“这是什么”，回答不了“这款饵为什么值得买”
- 不同饵型的关注点完全不同，硬饵看泳姿、潜深和重心系统，软饵看入数、材质和尾型，金属饵又更看下沉、闪烁和挂钩方案
- 真正能留住玩家的，不只是基础参数，而是“这款饵的小细节、小巧思、操作节奏、停顿表现和适配场景”

所以鱼饵字段设计必须同时服务两个层：

1. **决策主层**  
   先让用户看懂：饵型、水层、动作、尺寸、重量、潜深、浮力、钩规、入数这些决定“能不能买、适不适合我”的关键字段。
2. **深读层**  
   再让用户理解：这款饵的设计亮点、泳姿特征、停顿反馈、结构巧思、操作方式和适配水况这些真正构成“看点”的内容。

### 7.6 当前定义模糊字段

这些字段不是没有，而是当前语义、口径或覆盖还不稳：

- `type_tips`
  现在有时像“鱼饵子类”，有时像“玩法提示”，后续要明确它更偏标签还是说明。
- `system`
  当前已承接主筛选，但不同品牌和不同 detail 表的枚举口径还需要收敛。
- `action`
  当前既像“动作方式”，也可能承接“操作节奏”，后续需要明确是官方字段还是 GearSage 归纳。
- `depth`
  当前不同饵型可能是单值、范围或 marketing 文案，后续要配合 `depth_range`。
- `Description`
  当前应统一理解为主商品在官网的描述文字，属于补充层；若未来需要规格级说明，应拆出新字段，不再混用。
- `hook_size`
  不同饵型的写法差异较大，后续应配合 `hook_spec` 做归一。

## 九、第一版实施顺序建议

### 9.1 第一刀：补齐 detail / compare 真正缺的 P0 字段

优先顺序建议：

1. 渔轮：`spool_diameter_mm / spool_width_mm / spool_depth_normalized / brake_type_normalized / size_family / fit_style_tags`
2. 鱼竿：`power_normalized / action_normalized / tip_type / fit_style_tags`
3. 鱼饵：`buoyancy / depth / hook_size / quantity / fit_style_tags`

- `size_family`
- `power_normalized`
- `action_normalized`
- `tip_type`
- `depth_range`
- `hook_spec`

### 9.3 第三刀：继续做深读字段池，但不阻塞主链路

优先从渔轮 / 鱼竿 / 鱼饵各自挑 2~3 个高价值字段，进入人工维护或玩家补录池：

- 渔轮：`spool_weight_g / spool_axis_type / knob_size`

- 鱼饵：`design_highlights / swim_posture_note / retrieve_note`

## 十、当前明确不做

第一版不做：

1. 把所有 `deep_fields` 都推进 Excel 正式列  
2. 把品牌技术做成主对比字段  
3. 用爬虫直接输出“适合谁买”的结论  
4. 为了追求字段丰富度而破坏当前 `rate/excel` 基准稳定性  
5. 让前端继续直接从 `raw_json` 临时拼一堆没有统一口径的字段

## 十一、一句话口径

这份文档回答的是：

> **现在装备库该先推进哪些字段，它们当前是什么状态，该怎么进入系统。**

它是后续继续拆 Excel、导入脚本、接口返回结构、详情页和对比页字段排序的施工底表。
