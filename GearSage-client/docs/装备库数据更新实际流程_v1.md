## 1. 目标

这份文档只回答一个问题：

> 现在更新装备库数据，实际应该按什么顺序做。

它不是字段设计文档，也不是采集 SOP 的完整版。  
它的作用是给日常“更新数据 -> 上库 -> 客户端可见”这条链一个清晰入口。

## 2. 三层角色

### 2.1 中间数据层

位置：

- `/Users/tommy/GearSage/GearSage-client/pkgGear/data_raw/`

典型文件：

- `*_import.xlsx`
- `*_normalized.json`

作用：

- 这是采集、清洗、字段补充后的候选数据
- 不直接视为最终上线数据

常见问题：

- 主表拆分
- detail 归属错误
- detail id 冲突
- 描述、图片、字段覆盖不稳定

### 2.2 最终基准层

位置：

- `/Users/tommy/GearSage/GearSage-client/rate/excel/`

作用：

- 这是数据库导入唯一真实来源
- 是最终发布源，不是脚本随意覆盖的中间产物

当前规则：

- `rod/reel` 主表按字段级 merge
- detail 按 `gear_id` 切片同步
- 不允许整表覆盖
- 不允许空值覆盖最终表已有值
- `is_show` 始终以最终表为准

### 2.3 数据库发布层

位置：

- `/Users/tommy/GearSage/GearSage-api/scripts/import_gear_excel.js`

作用：

- 把 `rate/excel` 全量发布到数据库

行为：

- 先清空 `gear_brands / gear_master / gear_variants`
- 再按最终表全量重建

结论：

> 数据库不是手工维护源，只是 `rate/excel` 的发布结果。

## 4. 当前标准流程

### 第一步：生成或更新中间数据

输入：

- `normalized.json`
- 各类 `build_*_import.*`
- 各类 `to_excel_*`

输出：

- `pkgGear/data_raw/*_import.xlsx`

目标：

- 先把本次采集/补字段结果整理到中间层

### 第二步：同步到最终表

命令：

```bash
node /Users/tommy/GearSage/scripts/sync_rate_excel_from_imports.js
```

目标：

- 把中间层 merge 到 `rate/excel`

当前特性：

- `rod/reel` 主表字段级 merge
- detail 按 `gear_id` 切片同步
- 新品牌切片自动并入
- 无 detail 的旧主表会被清理
- 对已经全量重抓且中间表可信的 detail 切片，可在 `scripts/sync_rate_excel_from_imports.js` 中配置 `clearBlankFields`。启用后，中间表空值也会清理最终表旧值，并在同步后校验这些字段与中间表完全一致。
- 如果某个品牌切片的主表字段已经确认以中间表为准，可在对应 replacement 上配置 `importPriorityIfNonBlank`，避免 `fillOnly` 字段保留旧错位值。

新增或回填主表字段时，必须先确认同步链路认识这个字段：

1. `scripts/gear_export_schema.js` 的对应 master header 包含该字段。
2. `scripts/sync_rate_excel_from_imports.js` 的对应 master merge policy 把该字段列入 `importPriority`、`finalPriority` 或 `fillOnly`。
3. 字段语义上如果来自中间表人工补全，通常应放入 `importPriority`，并且只允许非空覆盖。

否则即使 `pkgGear/data_raw/*_import.xlsx` 已经有新字段，写回 `rate/excel` 时也可能被最终表表头裁掉。

### 第三步：导库前校验

命令：

```bash
cd /Users/tommy/GearSage/GearSage-api
npm run import:gear:check
```

当前会检查：

- 主外键关系
- 重复 key
- 数据库字段长度超限
- reel `type` 合法值
- 主表有、detail 缺

### 第四步：正式导库

命令：

```bash
cd /Users/tommy/GearSage/GearSage-api
npm run import:gear
```

结果：

- gear 三表全量重建
- 搜索索引自动同步到客户端

## 5. 什么时候不要直接导库

遇到这些情况，先停下：

1. `sync_rate_excel_from_imports.js` 报切片冲突
2. `import:gear:check` 报主表无 detail
3. `import:gear:check` 报字段长度超限
4. `rate/excel` 和 import 在主键/类型口径上不一致
5. 新增主表字段没有进入 `gear_export_schema.js` 或 sync merge policy
6. 枚举字段出现非标准值，例如少逗号、旧枚举、跨类型枚举混用

原则：

> 导库阶段只做发布，不在导库时临时修语义问题。

## 6. 本次 fit_style_tags 更新补充记录

2026-05-02 的 rod / reel `fit_style_tags` 全量更新确认了以下执行口径：

- `fit_style_tags` 是主表字段，必须进入 `rate/excel/reel.xlsx` 和 `rate/excel/rod.xlsx` 主表。
- reel 主表新增该字段时，需要同步更新 `scripts/gear_export_schema.js` 的 `reelMaster` header。
- rod / reel 从中间表同步到最终表时，`fit_style_tags` 必须列入 `scripts/sync_rate_excel_from_imports.js` 的 master merge `importPriority`。
- 导库前要单独校验枚举合法性；例如 `远投bass` 必须修为 `远投,bass`。
- `drum` 当前不参与 reel `fit_style_tags` 筛选，字段应保持为空。
- 如果全量同步其他品类时出现旧主表无 detail，先按同步规则清理孤儿主表，再跑 `import:gear:check`。

## 7. rod product_technical 更新补充记录

2026-05-06 的 rod `product_technical` 字段口径修订为 SKU 级：

- `product_technical` 是 `rod_detail` 子型号字段，承接当前 SKU 实际适用的官方技术名词合集。
- 字段位置：`*_rod_import.xlsx / rod_detail`，列位放在 `Description` 后、`Extra Spec 1` 前；`rod` 主表不维护该字段。
- 来源限制：只允许从品牌官网产品页、品牌官方技术页、官方 PDF/catalogue 获取；白名单站点、玩家资料、规格推断、跨品牌等价推断不得直接落表。
- 填写格式：多个技术名使用 ` / ` 分隔，例如 `X 导环 / CI4+`；可空，不确定时留空。
- 写入导入表前必须同步更新 `scripts/gear_export_schema.js` 的 `rodDetail` header。
- 从中间表同步到最终表时，`product_technical` 随 `rod_detail` 切片同步；不得列入 rod master merge policy。
- 校验要求：字段内容只写技术名，不写来源说明、营销长句、钓组/饵型、玩家判断或适用场景解释。
- 优先级：若子型号 `Description` 中存在官方 `TECHNOLOGY：...` 或 `搭載技術：...`，以该子型号描述为最高优先级；只有子型号描述没有技术清单时，才回落到页面级 `DAIWA 技術` 模块。
- 若官网技术模块存在“仅某规格”“除外规格”“全旋转式/全拋投式”等适用关系，必须按 SKU / TYPE / PIECES 规则拆开，不得把整页技术合集套给所有子型号。
- 如果官网明确某 SKU 带有或不带有某项技术，`product_technical` 必须体现这个差异；没有特殊标注时，才认为页面级技术默认适用该页面下所有 SKU。
