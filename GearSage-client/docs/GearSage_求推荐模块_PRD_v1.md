# GearSage 求推荐模块设计文档 v1

版本：v1.0  
状态：可直接交给 Codex 进入实现拆分  
适用范围：GearSage-client / GearSage-api / 后续文档回写  
更新时间：2026-03-31

---

## 一、文档目标

本文档用于定义 GearSage 在当前“讨论&提问”模式下新增 **求推荐** 子类型的最小可落地方案，并同时明确：

1. 发布模式字段、字段规则、预选项
2. 求推荐详情页结构
3. 求推荐回答模板结构、字段、预选项
4. 采纳、反馈、身份沉淀的最小闭环
5. 后端数据与接口的最小增量改造面
6. 前后端分阶段实施顺序

本文档不是泛泛脑暴，而是给 Codex / 后端 / 前端直接接手的执行口径。

---

## 二、当前基线与设计约束

### 2.1 当前已存在的提问字段

当前 `讨论&提问` 模式在 topic 侧已经存在以下字段：

- `questionType`
- `relatedGearCategory`
- `relatedGearModel`
- `relatedGearItemId`
- `quickReplyOnly`

当前列表接口 `GET /mini/topic/all` 也已支持按 `questionType` 过滤。

### 2.2 当前设计约束

本方案必须遵守以下约束：

1. **不新开独立大模式**，继续挂在现有 `讨论&提问` 下
2. **不推翻当前 `/mini/topic*` 主链路**
3. **尽量以增量字段和增量 UI 完成**
4. **前端先做可发、可答、可采纳、可反馈的闭环**
5. **后端优先保持兼容**，不要为求推荐重做整套 topic/comment 结构
6. **文档优先小而真**，不提前引入复杂推荐算法或“自动决策系统”承诺

### 2.3 本期目标

本期求推荐模块的目标不是“平台直接替用户做购买决定”，而是：

> 让用户更快结构化表达自己的纠结，让回答者更快给出有边界的建议，并让“问题 -> 建议 -> 采纳 -> 结果反馈”形成可沉淀案例。

## 三、整体产品定位

### 3.1 模块归属

- 一级模式：`讨论&提问`
- 二级子类型：`求推荐`（“求推荐”和“求避坑”合成一个标签，求推荐，并且把这个标签放到列表最前，默认为选中状态）
- 建议口径：`questionType = recommend`

### 3.2 与普通提问的差异

普通提问的核心是“我有一个问题”。

求推荐的核心是：

- 我准备花钱
- 我在纠结
- 我希望别人结合预算、场景、阶段、诉求给建议
- 我后续可能会采纳某条建议，并补结果反馈

### 3.3 本模块最小闭环

1. 用户发一条求推荐帖
2. 他人按模板回答
3. 楼主采纳一条回答
4. 楼主后续补结果反馈
5. 用户主页沉淀“回答数 / 被采纳数 / 已反馈数”等身份数据

## 四、发布模式设计

## 4.1 基本策略

求推荐继续走当前发帖主链路：

- `PUT /mini/topic` 保存草稿
- `POST /mini/topic` 发布

在 payload 中继续复用：

- `topicCategory = 讨论&提问`
- `questionType = recommend`

并新增一个轻量对象字段：

- `recommendMeta`

说明：

- 第一版建议前后端都接受 `recommendMeta` 为 JSON 对象
- 暂不要求一开始就把所有子字段拆成数据库独立列
- 只要检索、列表摘要、详情展示需要的关键字段能落位即可

---

## 4.2 发布字段总览

### A. 基础识别层

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `questionType` | 是 | string | 固定值 `recommend` |
| `title` | 是 | string | 标题 |
| `relatedGearCategory` | 是 | string | 主询问装备类型 |
| `relatedGearModel` | 否 | string | 兼容旧链路，可选 |
| `relatedGearItemId` | 否 | number | 若已绑定装备库项则传 |
| `quickReplyOnly` | 否 | boolean | 是否只看简短直接建议 |

### B. 求推荐核心层（放入 `recommendMeta`）

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `recommendIntent` | 是 | string | 求推荐意图 |
| `budgetRange` | 是 | string | 预算区间 |
| `budgetFlexible` | 否 | string | 预算弹性 |
| `targetFish` | 是 | string[] | 目标鱼，最多 3 个 |
| `useScene` | 是 | string[] | 使用场景，最多 2 个 |
| `carePriorities` | 是 | string[] | 最在意的点，最多 3 个 |
| `avoidPoints` | 否 | string[] | 不想要的点，最多 3 个 |
| `currentStage` | 是 | string | 当前阶段 |
| `currentGear` | 否 | string | 当前已有装备 |
| `candidateOptions` | 否 | array | 候选项，最多 3 个 |
| `usageFrequency` | 否 | string | 使用频率 |
| `coreQuestion` | 是 | string | 核心纠结点 |

### C. 自由表达层

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `content` | 否 | string | 自由补充说明 |
| `images` | 否 | string[] | 图片，最多 3 张 |

---

## 4.3 发布字段详细定义

### 4.3.1 `title`

**规则**

- 长度：8 ~ 40 字
- 支持系统默认生成，用户可改写
- 不建议空标题发布

**默认标题生成建议**

- `【求推荐】1500预算水库野河泛用纺车轮怎么选`
- `【求推荐】第一次配鲈鱼竿，预算1000内`
- `【求推荐】二选一：A和B哪个更适合野河翘嘴`

---

### 4.3.2 `relatedGearCategory`

**类型**

- 单选

**预选项**

- 鱼竿
- 渔轮
- 鱼饵
- 鱼线
- 钩子/配件
- 整套搭配
- 其他

**规则**

- 一个求推荐帖只允许一个主品类
- 同时问竿和轮时，建议选 `整套搭配`

---

### 4.3.3 `recommendIntent`

**类型**

- 单选

**预选项**

- 第一套入门
- 现有装备补强
- 老装备替换
- 想升级进阶
- 二选一 / 三选一
- 整套求配

**规则**

- 默认选中：`第一套入门`
- 若用户填写了多个候选产品，可自动高亮建议：`二选一 / 三选一`

---

### 4.3.4 `budgetRange`

**类型**

- 单选

**预选项**

- 300 以内
- 300~500
- 500~800
- 800~1200
- 1200~1800
- 1800~2500
- 2500~4000
- 4000+

**规则**

- 第一版不建议开放自由输入 min/max
- 档位化更利于低成本发布和后续筛选

---

### 4.3.5 `budgetFlexible`

**类型**

- 单选

**预选项**

- 预算基本卡死
- 可上浮一点
- 可接受二手
- 只看全新

**规则**

- 非必填
- 该字段用于帮助回答者判断“是否值得再上一个档位”

---

### 4.3.6 `targetFish`

**类型**

- 多选，最多 3 个

**预选项**

- 鲈鱼
- 翘嘴
- 鳜鱼
- 黑鱼
- 马口 / 溪流小型鱼
- 海鲈
- 综合泛用
- 其他（自填）

**规则**

- 至少选 1 个
- “综合泛用”可与 1 个具体对象鱼并存

---

### 4.3.7 `useScene`

**类型**

- 多选，最多 2 个

**预选项**

- 野河
- 水库
- 溪流
- 近海
- 城市河道
- 黑坑 / 管理场
- 综合不固定

**规则**

- 至少选 1 个
- 选“综合不固定”时不建议再多选太多场景

---

### 4.3.8 `carePriorities`

**类型**

- 多选，最多 3 个

**预选项**

- 性价比
- 泛用
- 轻量
- 远投
- 顺滑
- 耐用
- 新手友好
- 控鱼 / 腰力
- 细腻手感
- 做工颜值
- 保值

**规则**

- 至少选 1 个
- 超过 3 个会明显稀释重点，前端应限制

---

### 4.3.9 `avoidPoints`

**类型**

- 多选，最多 3 个

**预选项**

- 不想太重
- 不想太贵
- 不想太娇气
- 不想太专用
- 不想难上手
- 不想后期维护麻烦
- 不想太挑线 / 挑饵
- 其他（自填）

**规则**

- 非必填
- 这是很重要的反向字段，可帮助回答者不要一味往上推

---

### 4.3.10 `currentStage`

**类型**

- 单选

**预选项**

- 刚入门，基本没有装备
- 已有一套，想补一个空位
- 已有多套，想升级一项
- 旧装备用久了想替换

**规则**

- 必填
- 这是回答者判断“该不该一步到位”的重要背景字段

---

### 4.3.11 `currentGear`

**类型**

- 文本

**规则**

- 最多 80 字
- 提示语：`已有装备可简单写，如“ML竿 + 1000纺车轮”`

---

### 4.3.12 `candidateOptions`

**类型**

- 数组，最多 3 个

**单项结构建议**

```json
{
  "gearItemId": 0,
  "label": "",
  "source": "manual|gear_library"
}
```

**规则**

- 可手填
- 若能关联装备库，优先关联 `gearItemId`
- 第一版允许候选项不完全命中装备库

---

### 4.3.13 `usageFrequency`

**类型**

- 单选

**预选项**

- 出钓必备 / 高频使用
- 每周一次左右
- 每月几次
- 偶尔玩玩

---

### 4.3.14 `coreQuestion`

**类型**

- 文本

**规则**

- 长度：20 ~ 120 字
- 必填
- 这是求推荐帖最关键的自由文本字段

**提示语建议**

- `你最纠结的点是什么？比如：第一次买轮，怕买重了；或者A/B两个型号不知道差价值不值。`

---

### 4.3.15 `content`

**类型**

- 文本

**规则**

- 长度：0 ~ 300 字
- 非必填
- 定位为背景补充，不承担唯一信息源角色

---

### 4.3.16 `images`

**规则**

- 最多 3 张
- 非必填
- 可用于候选图、现有装备图、场景图

---

### 4.3.17 `quickReplyOnly`

**类型**

- boolean

**前端文案建议**

- `只想看简短直接建议`

**规则**

- 开启后，回答区默认优先鼓励使用简版回答模板
- 不阻止长回复，只影响默认引导

---

## 4.4 发布页交互顺序建议

建议按以下顺序引导，不要一开始先塞标题和长正文：

1. 你想问什么装备
2. 你现在属于哪种情况
3. 预算大概多少
4. 主要钓什么鱼
5. 常在哪钓
6. 你最在意什么
7. 你最纠结的点是什么
8. 有没有候选项
9. 要不要只看简短建议
10. 最后再补标题 / 正文 / 图片

说明：

- 这个顺序更符合“先收敛，再补充”的心智
- 对求推荐来说，比传统“标题 + 正文”更低压

## 五、求推荐详情页设计

## 5.1 目标

求推荐详情页的目标不是“看热闹”，而是：

> 让后来进入这条帖的人 10 秒内看懂：楼主想买什么、在纠结什么、最有帮助的回答是哪条、最终结果是什么。

---

## 5.2 页面结构

### A. 作者头部

展示：

- 头像
- 昵称
- 基础 tag
- 简介一句
- 发帖时间
- 轻量身份指标（可选）
  - 被采纳数
  - 获赞数
  - 长测评数

说明：

- 目的是让读者快速判断楼主大概是什么类型用户
- 不做花哨展示

---

### B. 标题区

展示：

- 固定子类型标识：`求推荐`
- 标题
- 状态文案：
  - `等待推荐中`
  - `已采纳 1 条回答`
  - `已反馈购买结果`
  - `问题已解决`

---

### C. 求推荐摘要卡

建议标题：`楼主当前情况`

展示字段：

- 装备类型
- 预算
- 对象鱼
- 场景
- 当前阶段
- 最在意什么
- 不想要什么
- 候选项

说明：

- 这是整个详情页最重要的一块
- 必须放在正文前
- 用标签卡片展示，不要像表单回显

---

### D. 核心纠结区

建议标题：`楼主最纠结的是`

展示字段：

- `coreQuestion`

说明：

- 单独强调，不要埋进正文

---

### E. 正文补充区

展示：

- `content`
- `images`

说明：

- 仅作为补充背景
- 不再承担唯一信息说明职责

---

### F. 回答区分层

必须分成三层：

1. 已采纳回答
2. 规范回答
3. 普通评论

#### 已采纳回答

若存在采纳，固定置顶并显示：

- `已采纳`
- 采纳时间
- 采纳人（楼主）
- 若后续有反馈，可显示：`楼主已补反馈`

#### 规范回答

排序建议：

1. 已采纳
2. 被楼主标记“有帮助”
3. 点赞高
4. 新近

#### 普通评论

保留，但视觉弱化。

用途：

- 追问
- 补充
- 轻讨论

---

### G. 回答输入区

必须有两个入口：

#### 入口 1：规范回答

主按钮文案：

- `写规范推荐`
- `按模板回答`

#### 入口 2：普通评论

次按钮文案：

- `补充评论`

---

### H. 结果反馈卡

当帖子已进入后续阶段时，底部增加 `结果反馈卡`：

展示：

- 最终采纳了谁的建议
- 最终买了什么 / 暂时没买
- 为什么这么选
- 当前感受
- 是否会补长测评

---

## 六、回答模板设计

## 6.1 整体原则

一个好的求推荐回答，至少要回答 5 件事：

1. 我的结论是什么
2. 我为什么这样建议
3. 我凭什么这样答
4. 为什么不优先推荐另一条路
5. 这条建议最大的风险是什么

说明：

- 不能只剩一句“我推荐 XX”
- 同时也不能把回答门槛做得太高

---

## 6.2 回答模板字段总览

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `answerConclusion` | 是 | string | 我的结论 |
| `recommendedOption` | 否 | array | 推荐方向 / 候选项 |
| `answerBasis` | 是 | string | 我凭什么这样答 |
| `myScene` | 否 | string[] | 我的常用场景 |
| `fitReasons` | 是 | string[] | 为什么适合楼主 |
| `freeReasonText` | 否 | string | 一句补充说明 |
| `notRecommendReasons` | 否 | string[] | 为什么不优先推荐另一条路 |
| `riskNote` | 否 | string | 风险提醒 |
| `budgetChangeAdvice` | 否 | string | 预算变化时我会怎么改 |

---

## 6.3 回答字段详细定义

### 6.3.1 `answerConclusion`

**类型**

- 单选

**预选项**

- 更推荐 A
- 更推荐 B
- 这预算建议先保守买
- 这预算建议再攒一点
- 现在不建议升级
- 还缺信息，建议补充

**规则**

- 如果楼主没填候选项，则不显示 A / B 这种文案，前端应切换为通用结论

---

### 6.3.2 `recommendedOption`

**类型**

- 数组，最多 3 个

**规则**

- 可关联装备库
- 也允许填文本方向，如“先看 1000 档泛用纺车轮”

---

### 6.3.3 `answerBasis`

**类型**

- 单选

**预选项**

- 我长期用过这个 / 这类产品
- 我短期用过
- 我没用过这款，但用过同定位多个产品
- 我主要按参数和定位给建议

**规则**

- 必填
- 必须在回答卡里展示
- 它不是削弱回答者，而是给读者判断边界

---

### 6.3.4 `myScene`

**类型**

- 多选，最多 2 个

**预选项**

- 野河
- 水库
- 溪流
- 近海
- 城市河道
- 综合

---

### 6.3.5 `fitReasons`

**类型**

- 多选，最多 3 个

**预选项**

- 更贴楼主预算
- 更适合楼主场景
- 更适合新手
- 泛用性更高
- 容错更高
- 升级空间更合理
- 配套成本更低
- 更耐用 / 更省心
- 更符合楼主当前阶段

**规则**

- 至少选 1 个
- 这是模板回答的主体

---

### 6.3.6 `freeReasonText`

**类型**

- 文本

**规则**

- 0 ~ 120 字
- 用于补一句最关键的人话判断

示例：

- `你现在不是差那点顺滑，反而更需要容错。`
- `你这个场景用不上那部分高配，钱先省下来更值。`

---

### 6.3.7 `notRecommendReasons`

**类型**

- 多选，最多 2 个

**预选项**

- 超预算
- 太专用
- 新手不友好
- 提升感知没那么大
- 对楼主场景利用率不高
- 后期成本更高
- 风险比收益更大

说明：

- 这个字段很重要，它能把回答从“推销”变成“判断”

---

### 6.3.8 `riskNote`

**类型**

- 单选 + 可补一句

**预选项**

- 重量 / 手感见仁见智
- 长期耐久还要看使用强度
- 这条建议更偏保守
- 这条建议更偏一步到位
- 如果你特别追求极致手感，这个未必满足
- 其他（自填）

---

### 6.3.9 `budgetChangeAdvice`

**类型**

- 单选

**预选项**

- 如果预算再高一点，我会换另一个方向
- 如果预算再低一点，我会优先保留核心性能
- 当前预算已经够，不建议再上
- 当前预算偏紧，建议先降低预期

---

## 6.4 回答模板展示结构

最终在详情页里，不要回显成表单，而要渲染成以下结构：

### 我的结论

更推荐 A / 先保守买 / 先别升级

### 我的参考背景

我长期用过同类产品，常钓水库 / 野河

### 为什么这么建议

更贴预算、更适合楼主场景、对新手更友好

### 不优先推荐另一条路的原因

太专用、对当前阶段提升不明显

### 要注意的风险

如果你特别追求极致轻量，这条建议未必最优

### 预算变化时我会怎么改

预算再高一点，我会换到另一个方向

---

## 七、采纳交互设计

## 7.1 基本规则

- 一条求推荐帖只允许采纳 **1 条主回答**
- 第一版建议只允许采纳 `规范回答`
- 采纳不等于关闭评论区

---

## 7.2 采纳按钮规则

- 只有楼主可见
- 只出现在规范回答卡上
- 普通评论第一版不开放采纳

---

## 7.3 采纳确认弹窗

文案建议：

`采纳后，这条回答会被置顶展示。你后续仍可补充购买反馈。`

按钮：

- 取消
- 确认采纳

---

## 7.4 采纳后即时变化

### 页面变化

- 该回答置顶到第一位
- 回答卡出现 `已采纳`
- 帖子状态从 `等待推荐中` 变成 `已采纳 1 条回答`

### 数据变化

建议记录：

- `acceptedAnswerId`
- `acceptedAt`
- `acceptedByUserId`
- `acceptStatus`

`acceptStatus` 建议值：

- `none`
- `accepted`
- `feedback_added`
- `closed_without_accept`

---

## 7.5 是否允许改采纳

第一版建议：

- **允许改一次**
- 限制在采纳后 48 小时内

原因：

- 用户可能冲动采纳
- 后续看到更好的回答有修正空间

若想进一步简化，第一版也可先不开放改采纳。

---

## 7.6 采纳后是否关闭讨论

不建议关闭。

采纳后仍允许：

- 继续补充建议
- 楼主追问
- 楼主后续反馈

详情页顶部可提示：

`楼主已采纳一条回答，仍可继续补充建议。`

---

## 八、结果反馈设计

## 8.1 目标

反馈的价值比“采纳”更高，因为它把一条求推荐帖从“问答”升级成“案例”。

### 反馈回答的问题

- 最终买了什么 / 没买
- 为什么这么选
- 当前感受如何
- 后续会不会补长测评

---

## 8.2 触发方式

建议两种：

1. 采纳后详情页常驻 `补充反馈` 按钮
2. 后续若有站内消息，可在 3 天 / 7 天后提醒

---

## 8.3 反馈字段

### `finalDecisionType`

**必填，单选**

预选项：

- 按采纳建议买了
- 参考采纳建议后买了别的
- 暂时没买
- 放弃这次升级 / 购买

---

### `finalProduct`

**选填**

- 文本或装备库关联项

---

### `decisionReason`

**必填，单选或最多 2 项**

预选项：

- 更符合预算
- 更符合场景
- 最终觉得更稳妥
- 最终觉得没必要上更高价位
- 看了大家建议后思路更清楚
- 现实原因暂时不买
- 其他（自填）

---

### `resultSatisfaction`

**选填，单选**

预选项：

- 很满意
- 基本满意
- 一般
- 有点后悔
- 还没真正开始用

---

### `feedbackText`

**选填，文本**

规则：

- 0 ~ 120 字

提示语：

- `一句话说说现在的感受，比如“最后没上更贵的，反而更适合我现在。”`

---

### `willPostLongReview`

**选填，单选**

预选项：

- 之后会补长测评
- 可能会补
- 暂时不会

说明：

- 这个字段可以把求推荐往长测评导流

---

## 8.4 反馈展示

不要把反馈埋进评论流，建议单独做 `结果反馈卡`。

展示：

- 最终结果
- 最终买了什么 / 暂时没买
- 为什么这么选
- 当前感受
- 楼主一句补充
- 是否会补长测评

---

本期明确不做：

1. 平台自动推荐装备
2. 大样本评分系统

4. 多条回答同时采纳

## 十五、一句话收束

求推荐模块第一版不是为了让 GearSage 直接替用户做决定，而是为了让用户更低成本地表达纠结、让回答者更高质量地给建议，并让这些建议逐步沉淀成可复用的真实案例。
