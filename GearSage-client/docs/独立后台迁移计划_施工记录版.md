# 二、当前结论摘要

当前项目已经正式确定长期目标为 App 化，但第一阶段仍以微信小程序联调和迁移为主：

- 第一阶段前端运行方式：**微信小程序**
- 后续容器方向：**FinClip**
- 产品长期形态：**App**
- 后端框架：**Node.js + NestJS**
- 数据库：**PostgreSQL**
- 运行环境：**腾讯云轻量服务器**
- 反向代理：**Nginx**
- 进程管理：**PM2**
- 当前 API 基础地址：**https://api.gearsage.club**
- 媒体存储：后续接入 **COS**
- 登录方案：**手机号验证码登录 + Access Token / Refresh Token**
- 第一阶段验证码策略：**固定验证码 `123456`**
- 当前策略：**不迁移历史用户和帖子数据，允许清空重建**
- 当前原则：**先做最小可用独立后台，再逐步替换小程序云开发能力**

## 7.2 帖子模块

### `GET /mini/topic/all`
说明：
- 获取已发布帖子列表
- 仅返回：
  - `status = 2`
  - `isDelete = 0`

### `GET /mini/topic`
参数：
- `topicId`

说明：
- 获取帖子详情
- 若已删除或不存在，返回 404 业务结果

### `GET /mini/topic/tmp`
参数：
- `userId`

说明：
- 获取用户最新一条草稿
- 条件：
  - `status = 0`
  - `isDelete = 0`
- 排序：
  - `updateTime DESC`

### `GET /mini/topic/mine`
参数：
- `userId`

说明：
- 获取该用户自己的帖子列表
- 返回：
  - 草稿
  - 待审核
  - 已发布
- 不返回已删除帖子

### `PUT /mini/topic`
说明：
- 保存草稿
- 支持：
  - 新建草稿
  - 更新草稿
- 保存后：
  - `status = 0`
  - `isDelete = 0`
  - `publishTime = null`

### `POST /mini/topic`
说明：
- 发布帖子
- 支持：
  - 直接发布新帖子
  - 将已有草稿发布
- 发布后：
  - `status = 1`
  - `publishTime = NOW()`
  - `isDelete = 0`

### `DELETE /mini/topic`
参数：
- `topicId`

说明：
- 软删除帖子
- 删除后：
  - `isDelete = 1`
  - `updateTime = NOW()`

### `POST /mini/topic/like`
说明：
- 点赞 / 取消点赞切换
- 行为：
  - 未点赞 → 新增点赞记录，`likeCount + 1`
  - 已点赞 → 删除点赞记录，`likeCount - 1`

# 十、迁移执行总原则

## 10.1 先做后端骨架，再做前端替换

原则：

1. 先把独立后台核心接口做出来
2. 再把前端从云开发调用切到后台接口
3. 最后清理云函数和云数据库直查

## 10.2 不做大爆炸式迁移

当前项目采用**分模块渐进迁移**：

- 先帖子模块
- 再评论
- 再鉴权
- 再用户
- 再上传
- 再标签/商城/任务
- 最后装备库

## 10.3 每做完一步，必须回写本文档

任何新增内容都必须同步更新：

- 已完成接口
- 已完成数据库表
- 已完成服务器变更
- 当前运行方式
- 下一步计划

# 十四、当前已知问题与经验记录

## 14.1 终端 heredoc 容易被打乱

当前施工过程中已经多次出现：

- `cat > file <<'EOF'` 输入中混入其他命令
- 返回结果和下一条命令串在一起
- `EOF` 未单独结束

这会导致：

- 文件内容污染
- 接口代码损坏
- 返回结果难以阅读

### 处理原则
以后写多行文件时必须遵守：

1. 从 `cat > xxx <<'EOF'` 开始整段复制
2. 中途不插其他命令
3. 最后一行必须是**单独的 `EOF`**

## 14.2 当前验证方式有效

当前最可靠的验证方式仍然是：

- `npm run build`
- `pm2 restart gearsage-api`
- `curl`
- `psql`

这套方式应继续作为后续标准验证流程。
