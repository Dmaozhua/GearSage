# 正式短信接入最小改造面

版本：v1.0  
状态：P3-A 方案已拍板 / 待代码实施  
适用对象：Tommy / Codex / 后端 / 前端 / 合规准备

---

## 一、文档目标

这份文档用于梳理 GearSage 从当前“固定测试验证码 `123456`”切换到“腾讯云短信正式验证码”的**最小改造面**。

目标不是先做复杂登录风控体系，而是：

> 在不推翻现有鉴权主链的前提下，把 `send-code -> login -> me -> refresh -> logout` 从测试模式切到正式短信模式。

---

## 二、当前现状

当前真实实现位于：

- [auth.controller.ts](/Users/tommy/GearSage/GearSage-api/src/modules/auth/auth.controller.ts)
- [auth.service.ts](/Users/tommy/GearSage/GearSage-api/src/modules/auth/auth.service.ts)

当前口径：

1. `POST /auth/send-code`
   - 直接返回固定验证码 `123456`
2. `POST /auth/login`
   - 只要传入 `123456` 就可登录
3. `GET /auth/me`
4. `POST /auth/refresh`
5. `POST /auth/logout`

小程序当前已稳定依赖这条链。

---

## 三、改造约束

正式短信接入必须满足：

1. 不改路径：
   - `POST /auth/send-code`
   - `POST /auth/login`
   - `GET /auth/me`
   - `POST /auth/refresh`
   - `POST /auth/logout`
2. 不推翻返回结构 `{ code, message, data }`
3. 不回退到微信登录 / 云开发
4. 尽量不要求前端大改

---

## 四、最小改造结论

第一版正式短信建议：

1. 继续保留 `AuthController` 路由不变
2. 在 `AuthService` 内部把验证码校验从“固定值”切到“短信码记录校验”
3. 新增一个最小短信发送服务，不拆独立认证系统
4. 先接腾讯云短信
5. 保留一个可控的 `SMS_TEST_MODE` 开关，便于本地联调

---

## 五、与当前 NestJS 结构一致的落点

当前项目的实现风格是：

- 业务目录在 `src/modules/*`
- `app.module.ts` 直接注册 provider / controller

因此第一版建议新增：

```text
src/modules/auth/
  auth.service.ts
  sms.service.ts
  sms.tencent.service.ts
  dto/
```

说明：

- 短信能力第一版只服务登录链
- 放在 `auth/` 下是当前最小改造面
- 暂不新开独立 `sms` 大模块

`app.module.ts` 继续直接注册：

- `SmsService`
- `TencentSmsService`

1. 发出去
2. 能校验

第一版可以暂不单独新增：

- 短信模板表

- 登录风控大盘表
- 多因子认证表

## 七、接口改造面

## 7.1 `POST /auth/send-code`

路径不变。

第一版改造为：

1. 校验手机号格式
2. 校验发送频率
3. 生成 6 位随机验证码
4. 存入 `auth_sms_codes`
5. 调腾讯云短信发送

返回结构仍为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "phone": "13800000000",
    "expiresIn": 300,
    "isTestCode": false
  }
}
```

### 本地测试模式

当 `SMS_TEST_MODE=true` 时，允许继续返回：

```json
{
  "phone": "13800000000",
  "code": "123456",
  "expiresIn": 300,
  "isTestCode": true
}
```

说明：

- 这能保证本地开发和微信审核准备并行
- 但正式环境必须关闭

## 7.2 `POST /auth/login`

路径不变。

第一版改造为：

1. 按手机号查最近一条未过期、未使用验证码
2. 比较验证码 hash
3. 命中后把该验证码标记为 `used`
4. 继续沿用当前用户创建 / token 签发逻辑

`refresh / logout / me`：

- 不需要改协议
- 继续沿用当前实现

## 八、服务端最小改造面

## 8.1 `auth.service.ts`

当前需要改的只有两块：

### `sendCode(phone)`

从：

- 固定返回 `123456`

变成：

- 生成验证码
- 走短信服务
- 存发送记录

### `login(dto)`

从：

- `dto.code === 123456`

变成：

- 校验 `auth_sms_codes`

其余逻辑尽量不动：

- `findOrCreateUser`
- `issueTokens`
- `refresh`
- `logout`
- `getMe`

## 8.2 建议新增 `sms.service.ts`

职责：

1. 生成验证码
2. 发送频控
3. 调用 provider
4. 写入 `auth_sms_codes`

建议方法：

- `sendLoginCode(phone, context)`
- `verifyLoginCode(phone, code)`

## 8.3 建议新增 `sms.tencent.service.ts`

职责：

1. 封装腾讯云短信 SDK / API
2. 统一返回发送结果

建议内部返回：

```ts
type SmsSendResult = {
  success: boolean;
  provider: 'tencent';
  requestId: string;
  raw: Record<string, any>;
};
```

## 九、控频与风控最小口径

第一版至少做以下 4 条：

1. 同手机号 60 秒内不可重复发送
2. 同手机号 1 小时内最多 5 次
3. 同手机号 24 小时内最多 10 次
4. 验证码有效期 5 分钟

可选附加：

1. 同 IP 1 小时内最多 20 次
2. 登录失败过多后短时冻结

第一版不要求：

- 图形验证码
- 设备指纹
- 行为风控平台

## 十、前端最小改造面

## 10.1 登录页

理论上最小改造可以接近 0。

原因：

- 路由不变
- 提交参数不变
- 成功返回结构不变

### 可能需要的唯一调整

如果当前登录页把 `send-code` 返回的测试码直接显示或依赖它调试，则需要加一条：

- 仅当 `isTestCode = true` 时展示返回 code

正式环境下不再展示验证码。

## 10.2 `services/auth.js` / `services/api.js`

当前无需因为短信接入改造 token 模型。

继续沿用：

- login
- me
- refresh
- logout

## 十三、当前一句话结论

正式短信接入的最小改造面，不是重做鉴权，而是：

> **保持当前 `/auth/send-code` `/auth/login` `/auth/me` `/auth/refresh` `/auth/logout` 全部不换口径，只把 `send-code + login` 的验证码来源从固定测试值切成“腾讯云短信 + 数据库验证码记录校验”。**
