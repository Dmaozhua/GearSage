## 1. 迁移目标

把装备库数据资产彻底移出 `GearSage-client`，避免小程序打包、上传和 Git 提交继续携带大体积数据文件。

迁移后，`GearSage-client` 只保留客户端运行必需内容：

- 页面、组件、样式、交互逻辑
- `pkgGear/searchData/defData.js` 前端筛选静态配置
- 暂时保留运行时搜索索引 `pkgGear/searchData/Data.js` 和 `data/gear-search-data.js`
- `pkgGear/utils/官网解释_text_simple*.js` 这类运行时技术解释数据

装备库维护源、抓取中间层、报告、备份统一进入：

```text
/Users/tommy/GearSage-data/
  rate/excel/
  rate/webp/
  data_raw/
  reports/
  backups/
```

数据资产仓库：

```text
repo: /Users/tommy/GearSage-data
remote: git@github.com:Dmaozhua/GearSage-data.git
branch: rel
role: 装备库数据资产仓库，不放入 GearSage-client 小程序工程
```

## 2. 当前问题

当前大体积目录仍位于小程序目录内：

```text
GearSage-client/pkgGear/data_raw/  # 约 129MB，抓取、OCR、normalized、import、报告、缓存混杂
GearSage-client/rate/              # 约 26MB，最终 Excel 基准与旧本地 webp
```

这会造成三类问题：

1. 微信开发者工具可能把非运行数据纳入包体扫描或上传判断。
2. Git 容易误提交 Excel、OCR、抓取缓存、报告、备份文件。
3. 数据生产路径和客户端运行路径混在一起，后续脚本越来越难维护。

迁移后的原则：

> `GearSage-client` 不再是装备库数据资产仓库，只是消费后端 API 和少量运行时配置的小程序客户端。

## 9. 风险和处理方式

### 风险 1：历史脚本继续写旧目录

- 可执行脚本中发现旧路径，必须改为统一路径工具。

### 风险 2：导库读取不到 Excel

处理：

- `GEAR_EXCEL_DIR` 必须显式指向 `/Users/tommy/GearSage-data/rate/excel`。
- `import:gear:check` 是正式导库前硬门槛。

处理：

- 迁移后不再把 `rate/webp` 当客户端静态资源。
- 导入数据应优先写入 CDN URL。
- 如果发现数据库仍返回 `/rate/webp/*`，先修导入图片归一逻辑，再发布。

### 风险 4：Git 删除误伤运行代码

- `git rm -r --cached` 只针对：
  - `GearSage-client/pkgGear/data_raw`
  - `GearSage-client/rate`
- 不对 `pkgGear/pages`、`pkgGear/components`、`pkgGear/utils` 执行批量删除。

### 风险 5：历史文档链接失效

处理：

- 不第一时间批量改所有历史流程文档。
- 新执行入口以本文档和 `装备库数据更新实际流程_v1.md` 为准。
