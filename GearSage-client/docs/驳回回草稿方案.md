# 驳回回草稿方案

版本：v1.0  
状态：待实施  
适用对象：Tommy / Codex / 后端 / 前端 / 审核状态机

## 一、文档目标

这份文档用于定义 GearSage 在当前审核体系下的：

> 帖子“驳回后返回草稿”最小落地方案。

目标不是现在就做复杂的多轮审核工作流，而是先解决当前一个明显问题：

- 用户发帖后，如果未通过审核，当前对用户来说几乎像“帖子消失了”
- 用户不知道哪里改，也没有再次编辑的入口
- 这会直接伤害信任感和可用性

因此需要一个最小改造：

> 帖子被驳回后，不直接等同于删除，而是回到草稿，让用户之后还能继续编辑再发。

## 二、为什么现在值得做

当前 GearSage 已经具备：
- 帖子草稿保存
- 发布
- 待审核
- 审核后台通过 / 驳回 / 下架 / 恢复显示

同时，当前帖子状态体系已经存在：
- `0` 草稿
- `1` 待审核
- `2` 已发布

这意味着：
- 你不是没有草稿系统
- 你只是还没把“驳回”接回草稿链路

所以这件事不是新增一个大系统，而是：

> 把审核驳回动作，合理接回现有草稿体系。

## 三、当前问题定义

当前如果帖子驳回后只是“用户看不到了”，会有几个问题：

1. 用户不知道帖子发生了什么
2. 用户没有明确的再次编辑入口
3. 平台会显得很粗暴
4. 驳回理由即便后台有，前台也接不住
5. 审核闭环虽然在后台成立，但用户体验不成立

## 四、拍板结论

### 4.1 驳回不等于删除
帖子被驳回后：
- 不直接删除
- 不彻底消失
- 不保持已发布态

而是：
- 返回草稿

### 4.2 下架不等于回草稿
“驳回”和“下架”语义不同：

驳回：
- 发生在审核环节
- 内容尚未正式通过 / 不适合直接展示
- 建议动作：回草稿

下架：
- 发生在内容曾经已发布之后
- 属于治理动作
- 建议动作：不自动回草稿

### 4.3 恢复显示仍保持现有语义
“恢复显示”继续表示：
- 下架后的内容重新恢复可见
- 回到 `status = 2 + isDelete = 0`

不和“驳回回草稿”混用。

## 五、建议状态流转

### 草稿保存
- `status = 0`

### 提交审核 / 发布后待审核
- `status = 1`

### 审核通过
- `status = 2`

### 审核驳回
- `status = 0`
- 同时回写驳回原因
- 保持草稿可再次编辑

### 已发布后下架
- 继续按现有治理语义处理
- 推荐保持：
  - `status = 2`
  - `isDelete = 1`

### 下架后恢复
- `status = 2`
- `isDelete = 0`

## 六、与现有后台动作的关系

### 6.1 帖子通过
保持不变：
- `status = 2`

### 6.2 帖子驳回
建议从当前“仅治理动作”改为：
- `status = 0`
- `isDelete = 0`
- 写驳回原因
- 写后台日志
- 写用户通知

这件事的本质：
后台驳回动作变成“把帖子退回为可继续编辑的草稿”。

### 6.3 帖子下架
保持独立：
- 不回草稿
- 仍是运营 / 治理动作

### 6.4 帖子恢复显示
保持独立：
- 仅针对下架后的内容恢复公开展示

## 七、最小数据改造建议

第一版可以尽量少加字段，但建议至少加一个“驳回原因承载”。

### 7.1 优先方案：在帖子表增加字段
建议在 `bz_mini_topic` 最小补：
- `reviewRemark TEXT NOT NULL DEFAULT ''`

或语义更清楚的：
- `rejectReason TEXT NOT NULL DEFAULT ''`

用途：
- 记录最近一次驳回原因
- 给前端消息中心 / 草稿页展示

### 7.2 备选方案：只写操作日志，不写帖子表
不推荐作为第一版唯一方案。

原因：
- 前端查一次驳回原因会被迫依赖后台日志
- 不利于小程序“我的草稿”直接展示
- 语义太绕

### 7.3 更稳的折中方案
第一版同时保留：
1. `admin_operation_logs`
2. `bz_mini_topic.rejectReason`

这样：
- 后台动作可追溯
- 前台能直接显示

## 八、前端最小改造建议

### 8.1 我的发布页 / 草稿页
当前建议：
- 驳回后的帖子，重新进入“草稿”范围
- 用户在草稿列表里能看到它

最低可见信息：
- 标题
- 更新时间
- 一条短提示：`该帖子未通过审核，请修改后重新发布`

### 8.2 发布页
当用户从草稿再次进入编辑页时：
- 能继续编辑正文 / 图片 / 标题
- 再次点击发布时重新进入审核链

### 8.3 提示文案
推荐：
- `该帖子未通过审核，请修改后重新发布`
- 如果有驳回原因：
  - `未通过原因：xxx`

不要把后台内部技术术语直接给用户。

## 九、消息系统的配合方式

“驳回回草稿”应和消息系统一起落地。

建议在驳回时同时写一条消息：
- `type = topic_rejected`

标题：
- `你的帖子未通过审核`

正文：
- `你发布的《xxx》未通过审核，已退回草稿，请修改后重新发布。`

如果有驳回原因：
- 放入 `extra.reason`
- 前端可选择展开显示

这样用户就不会出现“帖子怎么没了”的困惑。

## 十、当前不建议做的复杂版本

第一版不建议上来就做：
- 多轮驳回历史
- 草稿版本对比
- 驳回后锁图 / 锁字段
- 部分字段修改后自动局部重审
- 后台和前台双向留言讨论

当前最值钱的是：

> 驳回后，帖子还能回到草稿，用户知道原因，还能继续改。

## 十一、实施顺序建议

1. 先拍板驳回动作的状态流转
2. 给 `bz_mini_topic` 补最小驳回原因字段
3. 修改后台 `reject` 动作：
   - 改为回草稿
   - 写驳回原因
   - 写日志
4. 修改“我的发布 / 草稿”前端表现
5. 配合消息系统写 `topic_rejected` 通知
6. 最后做一次本地 smoke：
   - 发帖
   - 待审核
   - 驳回
   - 草稿回读
   - 修改后再次提交

## 十二、第一版 smoke 标准

一轮通过标准建议为：
1. 用户发一条帖子
2. 帖子进入待审核
3. 后台执行驳回
4. 用户在小程序里能看到该帖子重新回到草稿
5. 草稿能继续打开编辑
6. 草稿再次发布后能重新进入审核
7. 用户收到驳回通知
8. 如果写了驳回原因，前台能读到

## 十三、与现有治理语义的边界

要明确区分这两件事：

驳回
- 本质：审核未通过
- 用户侧处理：回草稿

下架
- 本质：治理动作
- 用户侧处理：通知下架，不自动回草稿

如果把两者混成一个动作，后面会很乱。

## 十四、一句话结论

“驳回回草稿”不是额外做复杂审核系统，而是：

> 把当前已经存在的草稿能力接回审核驳回链路，让帖子未通过审核时对用户可解释、可修改、可再次提交。
