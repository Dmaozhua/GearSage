# 小程序脱离微信云开发迁移计划

## 文档目的

这份文档用于同时承担两件事：

1. 作为当前代码现状的盘点结果，回答迁移前需要确认的几个关键问题。
2. 作为后续迁移执行的进度看板，方便持续勾选和补充。

## 当前结论摘要

当前项目不是“纯微信云开发”，而是一个**独立后台 + 微信云开发混用**的状态：

- 已经存在独立后台请求入口，当前标准生产域名已统一为 `https://api.gearsage.club`。
- 但 `services/api.js` 内部又把一批 `/mini/*` 路由优先映射到 `wx.cloud.callFunction('miniApi')`。
- 图片上传、云文件下载、临时访问链接、装备库数据读取等，仍然明显依赖微信云开发。

这意味着本次迁移不能只改域名，而是需要把“云函数路由层、云数据库访问层、云存储访问层”一起拆掉。

## 一、现在哪些功能还依赖 `wx.cloud` / 云函数 / 云数据库

### 1. 全局初始化层

- `app.js`
  - 启动时执行 `wx.cloud.init(...)`
  - 说明整个小程序仍然把云开发环境视为可用基础设施

### 2. 请求层仍依赖云函数

- `services/api.js`
  - 定义了 `CLOUD_ACTION_MAP`
  - 当前至少有 26 个 `/mini/*` 接口会优先转成 `wx.cloud.callFunction({ name: 'miniApi' })`
  - 只有未命中映射的接口才会继续走 `wx.request` 到独立后台

当前映射到云函数的主要业务包括：

- 用户
  - `/mini/user/login`
  - `/mini/user/info`
  - `/mini/user/update`
  - `/mini/user/points`
- 邀请
  - `/mini/invite/bind`
  - `/mini/invite/summary`
- 帖子
  - `/mini/topic`
  - `/mini/topic/all`
  - `/mini/topic/mine`
  - `/mini/topic/tmp`
  - `/mini/topic/like`
- 评论
  - `/mini/comment`
- 任务/成就
  - `/mini/taskFeat`
  - `/mini/taskFeat/unfinish`
- 标签
  - `/mini/tag/batch`
  - `/mini/tag/usable`
  - `/mini/tag/used`
- 商品/积分商城
  - `/mini/goods`

结论：

- 这些业务虽然代码写成了 `api.get/post(...)`，但实际仍未真正脱离云函数。
- `services/api.js` 是本次迁移的关键切口之一。

### 3. 现存云函数

当前 `cloudfunctions/` 下仍有以下函数或适配代码：

- `miniApi`
  - 主业务云函数
  - 文件体量很大，约 2814 行
  - 承载用户、帖子、评论、标签、任务、商品、装备详情等核心逻辑
- `login`
  - 用于获取 `openid`
- `mergeFileChunks`
  - 用于云存储分片文件合并
- `fixUserTags`
  - 一次性数据修复脚本，直接操作云数据库
- `miniprogram-adapter/api.js`
  - 一个把 REST 调用改成 `wx.cloud.callFunction` 的适配层
- `pkgLabs/test-cloud`
  - 云函数测试页

迁移判断：

- `miniApi` 必须迁出到独立后台
- `login` 必须并入独立后台的手机号登录与鉴权链路，最终下线
- `mergeFileChunks` 需要改造成独立后台/对象存储分片上传方案，或直接废弃
- `fixUserTags` 需要改造成一次性后端迁移脚本

这些页面主链路基本已经是 API 驱动，迁移成本相对低：

- `pages/index/index.js`
  - 首页帖子流主要走 `apiService.getAllTopics()` 等接口
- `pkgContent/detail/detail.js`
  - 帖子详情、评论、点赞主要走 `services/api.js`
- `pages/points/points.js`
  - 积分/任务主要走 `services/api.js`

注意：

- 但从页面代码层面看，这些页面本身通常不需要重做 UI，只需要改请求落点。

#### 3. `services/api.js` 的非云函数部分

可以保留的部分：

- 请求封装结构
- token 注入
- 重试/去重/离线队列思路
- `wx.request` 走独立域名的流程
- `/upload` 直传后台的能力

### 必须重写或重构的部分

#### 1. `services/api.js` 里的云函数分流

必须改掉：

- `CLOUD_ACTION_MAP`
- 命中 `/mini/*` 后优先 `wx.cloud.callFunction('miniApi')` 的逻辑

目标应变为：

- 所有业务统一直连独立后台
- 不再存在“看上去是 REST，实际转云函数”的隐藏分支

#### 2. `miniApi` 云函数的全部业务逻辑

`cloudfunctions/miniApi/index.js` 是这次迁移最大的后端工作量来源。

需要迁出的能力包括：

- 登录与用户信息
- 邀请绑定
- 帖子草稿/发布/删除/详情/列表/我的帖子/点赞
- 评论列表/发表评论
- 标签衣橱/佩戴配置
- 商品列表/兑换
- 任务与成就
- 装备详情接口

建议目标：

- 拆到独立后台的 controller/service/repository
- 同时补充接口契约和数据库表结构映射

#### 3. 所有云数据库直查逻辑

必须改成独立后台接口的模块：

- `pkgGear/pages/list/list.js`
- `pkgGear/pages/detail/detail.js`
- `pkgContent/edit-profile/cloudDB.js`
- `pkgContent/publish/cloudDB.js`
- `pkgContent/edit-profile/edit-profile.js` 中残留的云数据库代码

#### 4. 所有云存储读写逻辑

必须重写：

- `wx.cloud.uploadFile`
- `wx.cloud.downloadFile`
- `wx.cloud.getTempFileURL`
- `cloud://` 文件 ID 判断与处理

建议目标：

- 后台下发可访问 URL，前端直接使用 URL
- 上传改为：
  - 后台中转上传，或
  - 对象存储预签名上传，或
  - COS 临时密钥/分片上传

#### 5. 媒体 URL 模型

当前前端很多地方默认：

- 图片可能是 `cloud://...`
- 需要额外换成临时 URL

迁移后建议统一为：

- 数据库存永久访问 URL，或
- 数据库存对象 key + 由后端统一签名/拼接 URL

否则前端会一直背着 `tempUrlManager` 这类只适用于云开发的历史包袱。

#### 6. 登录链路重写

当前状态：

- 前端 `services/auth.js` 仍使用 `wx.login`
- 云函数 `login` 可获取 `openid`
- `miniApi` 中也依赖 `wxContext`

迁移后目标：

- 前端改为手机号验证码登录
- 认证接口统一走 `/auth/*`
- 第一阶段固定验证码 `123456`
- 独立后台生成自己的 accessToken / refreshToken
- 微信 `login` 云函数下线

结论：

- `wx.login` 不再作为主链路
- `login` 云函数必须下线
- `services/auth.js` 需要重写为手机号登录模型

## 七、已确认的风险

### 风险 1：表面是 REST，实际仍走云函数

风险说明：

- 页面代码大量使用 `services/api.js`
- 但 `/mini/*` 业务并没有真正落到独立后台

影响：

- 如果只改页面调用，不改 `services/api.js`，迁移会出现“看起来改了，实际上没切走”的假象

### 风险 2：图片历史数据仍是 `cloud://`

风险说明：

- 首页头像
- 个人资料头像/背景
- 发帖图片
- 购买凭证

都存在 `cloud://` 资源引用逻辑

影响：

- 新老数据会并存
- 前端会长期保留兼容逻辑
- 不提前设计回填方案，后面会越来越难清理

### 风险 3：装备库完全没有抽象层

风险说明：

- `pkgGear/pages/list/list.js`
- `pkgGear/pages/detail/detail.js`

直接拿 `wx.cloud.database()` 查询

影响：

- 这部分不能只改接口地址
- 需要先定义后端查询接口和筛选参数

### 风险 4：存在残留旧实现，容易误导

典型例子：

- `pkgContent/edit-profile/edit-profile.js`
  - 主流程已在用 `api.updateUserInfo`
  - 但仍保留 `saveUserInfoToCloud()` 和 `getUserOpenId()`
- `pkgContent/publish/publish.js`
  - 主流程已在用 `apiService.createTopic/publishTopic`
  - 但仍然依赖云存储上传
  - 还残留 `require('./cloudDB')`

影响：

- 如果不先做清理，后续很容易在改造时误走旧链路
