## 二、当前基线与约束

### 2.1 当前已经存在的基础能力

当前已明确存在的接口与链路：

- 用户信息接口：`GET /mini/user/info?id=:id`
- 帖子主链路：`GET /mini/topic/all`、`GET /mini/topic`、`PUT /mini/topic`、`POST /mini/topic`
- 评论主链路：`GET /mini/comment`、`PUT /mini/comment`、`POST /mini/comment/like`、`DELETE /mini/comment`
- 求推荐采纳：`POST /mini/topic/recommend/accept`
- 装备库：
  - `GET /mini/gear/brands`
  - `GET /mini/gear/list`
  - `GET /mini/gear/detail`

### 2.2 当前已经存在的求推荐能力

当前求推荐第一刀已经落地的部分：

- `questionType = recommend`
- `recommendMeta` 作为 topic 增量 JSON 字段
- `commentType = recommend_answer`
- `recommendAnswerMeta` 作为评论增量 JSON 字段
- 楼主可采纳 1 条规范回答
- 已采纳回答可从规范回答区抽出并置顶
- 帖子头部可显示 `等待推荐中 / 已采纳 1 条回答`

### 2.3 本文档必须遵守的约束

1. 不推翻当前 `/mini/topic*`、`/mini/comment*`、`/mini/gear/*` 主链路
2. 不把“他人主页”做成大而全社交主页
3. 不把“被采纳统计”做成复杂答主等级系统
4. 不把“装备库与求推荐联动”做成自动推荐算法
5. 继续优先采用增量字段、增量页面、增量接口
6. 前后端都保持对当前 NestJS + PostgreSQL + 小程序链路的兼容

## 三、整体目标

这一层的目标不是做热闹，而是让 GearSage 多出三种真实价值：

### 3.1 让用户更快判断“这个人值不值得信”

也就是：

- 他是谁
- 他大概是什么类型的玩家
- 他在 GearSage 里主要做什么
- 他给过多少有价值的建议

### 3.2 让“被采纳”成为真正有沉淀价值的身份信号

不是虚荣数字，而是：

- 你认真答过多少求推荐
- 你的回答被楼主采纳过多少次
- 你是不是持续提供有帮助的建议

### 3.3 让装备库不只是“看参数”，还能顺滑进入“我这种情况该怎么选”

也就是：

- 在装备详情页看完后，能继续去发求推荐
- 在未来对比页纠结时，能带着候选项去发求推荐
- 求推荐结果未来也能反向沉淀回装备库语境

## 四、他人主页设计

## 4.1 产品定位

他人主页不是“个人秀场”，而是：

> 一个帮助用户快速判断对方经验画像、贡献类型、求推荐回答可信度的最小资料页。

它的核心问题不是“这个人有多酷”，而是：

- 他是不是和我像
- 他是不是值得我继续看
- 他主要擅长发测评，还是擅长回答求推荐

---

## 4.2 页面入口

建议入口统一为：

- 帖子详情页点击作者头像 / 昵称进入
- 评论区点击回答者头像 / 昵称进入
- 求推荐帖中点击被采纳回答的作者进入

第一版不必新增“关注”体系，也不必做复杂跳转栈。

---

## 4.3 页面结构

### A. 头部基础信息区

展示：

- 头像
- 昵称
- 背景图
- 简介
- 基础 tag
- 所在地区（若后续需要，仅展示模糊信息，不展示精确位置）

说明：

- 当前目标不是暴露个人隐私，而是建立最基本的“人”的存在感
- 如果用户未设置背景图，走默认背景图即可

### B. 经验画像区

建议新增轻量字段展示：

- 常玩场景：如 水库 / 野河 / 溪流 / 近海
- 主要对象鱼：如 鲈鱼 / 翘嘴 / 鳜鱼 / 综合泛用
- 当前偏好装备方向：可后续从内容侧反推，第一版可不做

说明：

- 这一块的价值很高，因为它比“粉丝数”更适合 GearSage
- 但第一版不强求所有字段都来自真实结构化资料
- 没填就不展示，不做占位羞辱

### C. 论坛贡献区

这是他人主页最重要的功能区。

建议展示以下指标：

- 发帖数
- 长测评数
- 求推荐回答数
- 被采纳数
- 获赞数

说明：

- 这些指标的目的不是炫耀，而是帮助别人判断“这个人在 GearSage 里主要是做什么的”
- 第一版不建议展示复杂比率和算法分

### D. 最近代表内容区

建议分成两块：

#### D1. 最近长测评 / 代表帖子

展示：

- 最近 3 条高质量帖子
- 优先展示长测评 / 好物分享
- 若没有，则展示最近帖子

#### D2. 最近被采纳回答

展示：

- 最近 3 条被采纳的求推荐回答
- 卡片上直接展示：
  - 回答结论
  - 适用场景
  - 被采纳时间
  - 原帖标题

说明：

- 这块是“他人主页”与求推荐系统真正连起来的核心区域
- 它比单纯显示一个数字更有信任感

---

- `background`

### 求推荐相关统计字段

- `recommendAnswerCount`
- `acceptedAnswerCount`
- `recommendPostCount`
- `recommendSolvedCount`
- `recommendFeedbackCount`

### 通用贡献字段

- `topicCount`
- `longReviewCount`
- `likeReceivedCount`

### 代表内容字段

- `recentTopics`
- `recentAcceptedAnswers`

---

## 4.5 展示规则

### 必须展示

- 头像
- 昵称
- 简介
- 基础 tag
- 发帖数 / 被采纳数 / 求推荐回答数（至少其一）

### 有值才展示

- 背景图
- 常玩场景
- 主要对象鱼
- 最近被采纳回答
- 最近长测评

### 第一版不展示

- 粉丝数
- 关注数
- 连续登录天数
- 复杂成长等级
- “答主等级 S/A/B”
- “采纳率百分比”

原因：

- 这些会把 GearSage 过早推向“泛社区炫耀结构”
- 当前更重要的是建立可信、可读、克制的身份感

## 五、被采纳统计设计

## 5.1 目标

被采纳统计不是为了游戏化，而是为了让求推荐回答逐步长出“可被信”的沉淀。

它回答的是：

- 这个人认真回答过多少次
- 他的建议被楼主采纳过多少次
- 楼主是否后续补了反馈

---

## 5.2 指标定义

### `recommendAnswerCount`

定义：

- 用户发表的 `commentType = recommend_answer` 总数

用途：

- 表示这个用户在 GearSage 里认真回答求推荐的次数

### `acceptedAnswerCount`

定义：

- 用户发表的 `recommend_answer` 中，被楼主采纳为主回答的次数

用途：

- 这是最核心的身份沉淀指标

### `recommendPostCount`

定义：

- 用户发表的 `questionType = recommend` 帖子数

用途：

- 表示这个人有多少次主动发起求推荐

### `recommendSolvedCount`

定义：

- 用户发起的求推荐帖中，已存在采纳主回答的帖子数

用途：

- 用来表示“已解决求推荐”的沉淀程度

### `recommendFeedbackCount`

定义：

- 用户发起的求推荐帖中，已补充结果反馈的帖子数

用途：

- 表示这个用户是否愿意把问题变成完整案例

---

## 5.3 统计展示规则

### 他人主页展示建议

建议第一版展示：

- 求推荐回答：`X`
- 被采纳：`Y`
- 发起求推荐：`Z`

如需再加一项，优先加：

- 已补反馈：`N`

### 帖子详情作者头部展示建议

在求推荐帖详情页作者头部，可轻量展示：

- 被采纳数
- 获赞数
- 长测评数

### 回答卡作者信息展示建议

在规范回答卡中，回答者昵称下可轻量展示：

- `被采纳 X 次`
- 或 `回答 X / 被采纳 Y`

注意：

- 这里不要展示复杂 badge 墙
- 文案越简单越可信

---

## 5.4 第一版不建议展示的衍生指标

以下指标可以先记不显：

- 采纳率
- 回答平均点赞数
- 回答平均被标记“有帮助”次数
- 被采纳后真实执行率
- 回答者场景命中率

原因：

- 当前样本太小，容易误导
- 前期直接展示百分比会让数据很难看，也不稳定

---

## 5.5 数据来源建议

### 当前最小口径

第一版不必为了统计重做单独的统计表。

可以基于：

- `bz_topic_comment.commentType`
- `bz_mini_topic.extra.acceptedAnswerId`
- `questionType = recommend`
- 后续反馈字段

在查询用户详情时聚合得到。

### 后续可优化方向

如果后续性能或复杂度上来，再考虑：

- 用户统计冗余字段
- 异步回写用户统计表
- 每日归档统计任务

第一版不必提前做重。

## 六、求推荐与装备库详情页 / 对比页跳转关系

## 6.1 整体目标

装备库当前已经完成：

- 列表页切后台
- 详情页切后台
- 相关推荐切后台
- 详情页可通过 `GET /mini/topic/all` 拉关联帖子

这一层新增的目标是：

> 让用户在“看参数 / 看详情 / 看完还纠结”的时候，能自然进入求推荐，而不是退出 GearSage 去外部问。

---

## 6.2 三条关系链

### 链路 A：装备详情页 -> 发求推荐

适用场景：

- 用户看完某个具体产品详情，仍然不确定自己适不适合买

建议新增按钮：

- `我这种情况适合吗？去求推荐`
- 或更短：`去求推荐`

点击后跳转到 `讨论&提问 -> 求推荐` 发布页，并自动预填：

- `relatedGearCategory`
- `relatedGearItemId`
- `candidateOptions[0]`
- 若当前 gear 详情已有主品类 / 品牌 / 型号，可顺带填入候选项 label

示例预填：

```json
{
  "questionType": "recommend",
  "relatedGearCategory": "reel",
  "relatedGearItemId": 123,
  "recommendMeta": {
    "candidateOptions": [
      {
        "gearItemId": 123,
        "label": "XX 纺车轮",
        "source": "gear_library"
      }
    ]
  }
}
```

说明：

- 这是最小且最真实的联动
- 不需要自动替用户做决定，只需要把“当前正在看这个产品”带过去

---

### 链路 B：装备对比页 -> 发求推荐

适用场景：

- 用户已经有 2~3 个候选项，进入 GearSage 对比页对比完之后，还是拿不准

建议新增按钮：

- `还是纠结？带着这几个候选去求推荐`

点击后跳转到求推荐发布页，并自动预填：

- `recommendIntent = compare_options`
- `relatedGearCategory`
- `candidateOptions`
- 可选补一个默认 `coreQuestion` 引导语

示例：

```json
{
  "questionType": "recommend",
  "relatedGearCategory": "rod",
  "recommendMeta": {
    "recommendIntent": "compare_options",
    "candidateOptions": [
      { "gearItemId": 11, "label": "A 竿", "source": "gear_library" },
      { "gearItemId": 27, "label": "B 竿", "source": "gear_library" }
    ],
    "coreQuestion": "这两个我已经对比过参数了，但还是不确定哪个更适合我当前场景。"
  }
}
```

说明：

- 当前“装备对比页”还没有正式单独立项实现，但这个跳转口径应先拍板
- 后续无论对比页先做成独立页面还是详情页里的并排比较模块，都可复用这一跳转协议

---

### 链路 C：求推荐结果 -> 反向沉淀到装备语境

适用场景：

- 楼主补了结果反馈
- 最终买了某个具体产品，或明确放弃了某个产品方向

第一版暂不要求复杂智能回流。

第一版只建议做到：

- 若反馈中 `finalProduct` 绑定了 `gearItemId`
- 后续在该 gear 详情页的关联帖子区域可把这条求推荐帖自然命中

也就是说：

- 当前还是通过 `GET /mini/topic/all?gearItemId=...` 这条逻辑兜住
- 不新增复杂推荐图谱

---

## 6.3 装备详情页当前应展示什么跳转

### 装备详情页按钮建议

建议顺序：

1. 看参数 / 型号
2. 看关联帖子
3. 去求推荐

文案建议：

- `看完参数还是纠结？去求推荐`
- `带着这款装备去问问别人`

不建议文案：

- `智能推荐`
- `AI 替你选`
- `一键决策`

原因：

- 这会给 GearSage 现在还不具备的能力背书

---

## 6.4 对比页建议定位

对比页当前最合适的定位不是“自动告诉你谁更好”，而是：

> 帮你把差异看清，再把这些候选项顺滑带到求推荐。

所以对比页底部最合适的按钮就是：

- `继续求推荐`
- `带着这几个候选去问`

而不是：

- `生成购买结论`
- `推荐最佳答案`

---

## 6.5 跳转参数建议

为了减少前后端自由发挥，建议统一约定以下跳转参数：

### 从装备详情页进入求推荐页

- `from=gear_detail`
- `gearCategory`
- `gearItemId`
- `gearLabel`

### 从装备对比页进入求推荐页

- `from=gear_compare`
- `gearCategory`
- `candidateOptions`（JSON 字符串或本地缓存中转 key）

### 前端接收后映射到发布页默认值

- `relatedGearCategory`
- `relatedGearItemId`
- `recommendMeta.candidateOptions`
- `recommendMeta.recommendIntent`

- `pkgContent/user-profile/` 或 `pkgContent/other-user/`

本期明确不做：

1. 关注 / 粉丝系统
2. 私信系统
3. 复杂答主等级体系
4. 动态流式个人主页
5. 求推荐与装备库的自动算法推荐

- 当前目标是让 GearSage 多出更强的“人”的可信度与“装备库 -> 求推荐”的闭环
