# 装备库迁移拆分方案

更新时间：2026-04-02

## 一、目标

本方案用于把当前仍依赖微信云开发的装备库模块拆分成可逐步迁移的任务，而不是一次性整块重写。

本轮目标不是“立即迁完整个装备库”，而是输出可执行拆分：

- 页面依赖列表
- 云开发残留点
- 建议后端接口清单
- 数据库迁移建议
- 前后端切换顺序

### 3.4 数据采集与导入脚本

当前装备库源数据链路已从“纯人工 Excel 维护”升级为“**Scrapling 爬虫 -> 自动化枚举与分表 (Node.js) -> PostgreSQL 导入**”的工业化流程 (v2.2)：

**详细的采集、清洗、字段映射规则已沉淀至规范文档：[装备数据采集与清洗规范](装备数据采集与清洗规范.md)。**

- **自动化采集层 (`scripts/scrapers/`)**:
  - 使用 Python + Scrapling MCP，已完成 Daiwa, Shimano, Megabass 的渔轮、鱼竿、假饵 (Lure) 数据全量抓取。
  - 支持多线程并发、反爬代理与多语言描述回退策略。
- **自动清洗与分类 (`scripts/to_excel_*.js`)**:
  - `normalized.json` 中间产物通过 JS 脚本执行 `classifyLure` / `classifyRod`。
  - 自动完成 `system`, `water_column`, `action`, `type_tips` 枚举映射。
  - 自动输出分片表结构（如 hardbait, soft, metal 等）至 `data_raw/` 目录。
- **最终导入数据库 (`import_gear_excel.js`)**:
  - 负责把整理好的 `rate/excel` 导入 PostgreSQL。
  - 导入目标表：
    - `gear_brands`
    - `gear_master`
    - `gear_variants`
  - 当前命令：`npm run import:gear`
  - 同步重建本地搜索索引 `pkgGear/searchData/Data.js`。

## 四、建议后端接口清单

建议在 `GearSage-api` 新增以下接口：

- `GET /mini/gear/brands?type=reels|rods|lures`

返回：

- `id`
- `name`

### 4.2 装备列表

- `GET /mini/gear/list`

建议 query：

- `type`
- `page`
- `pageSize`
- `keyword`
- `brand`
- `system`
- `action`
- `waterColumn`
- `options`
- `brakeSys`

### 4.3 装备详情

- `GET /mini/gear/detail?id=:id&type=:type`

建议返回：

- 主记录
- 变体列表
- 品牌信息
- 当前 compare v1 继续复用此接口，不额外新增 compare endpoint
- 当前允许增量补：
  - `official_specs`
  - `gsc_traits`
  - `compare_profile`

### 4.4 关联帖子

优先复用现有帖子接口，不单独新建：

- `GET /mini/topic/all`

建议继续支持以下筛选：

- `topicCategory`
- `gearCategory`
- `gearModel`
- `gearItemId`

## 五、数据库迁移建议

当前 GearSage-api 已落地统一 gear 表结构：

- `gear_brands`
- `gear_master`
- `gear_variants`

当前设计取舍：

- 用统一主表 + 统一变体表承接原 `bz_rate_*`
- 保留 `raw_json`，降低 Excel 字段变化时的维护成本
- gear 接口当前优先读 PostgreSQL，只有未导入时才回退 Excel

## 六、建议迁移顺序

### 阶段 1

- 后端先补品牌接口
- 后端补装备列表接口
- 前端列表页先从云数据库切到 HTTP
- 当前状态：已完成

### 阶段 2

- 后端补装备详情接口
- 后端补变体明细返回
- 前端详情页从云数据库切到 HTTP
- 当前状态：已完成

### 阶段 3

- 关联帖子改为复用独立后台帖子接口
- 清理详情页 `wx.cloud.callFunction('topic.all')`
- 当前状态：已完成

### 阶段 4

- 清理 `gear.detail` 云函数依赖
- 清理导入脚本中的 `cloud://` 路径
- `Excel -> PostgreSQL` 导入链落地
- 当前状态：已完成第一版

## 七、当前结论

- 装备库主页面已经完成第一版切后台
- 装备库当前已补“详情页重组 + 对比页 v1（前端聚合版）”
- 当前剩余问题不再是“小程序是否直连云数据库”，而是：
  - 数据源已全面升级为 `爬虫抓取 -> normalized.json -> Excel -> PostgreSQL` 的自动化链路，人工维护工作量大幅降低。
  - 需要维护多套品牌和品类（如 Megabass, Daiwa, Shimano）的 `classifyLure`/`classifyRod` 枚举映射规则，以保证导入数据的可用性。
  - 本地 `searchData/Data.js` 当前已作为搜索辅助索引继续保留
  - 列表页当前采用“后端筛选 + 前端本地索引收敛”的混合策略
  - 推荐家族当前已区分“有覆盖 / 待补”，可避免把未覆盖词误当成功能异常
  - 更完整的筛选字段与搜索优化仍待继续补全
  - `gsc_traits / compare_profile` 当前仍以前端聚合与文案提示为主，后续再决定是否下沉到后端返回
