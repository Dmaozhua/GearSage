# 装备库迁移拆分方案

更新时间：2026-03-25

## 一、目标

本方案用于把当前仍依赖微信云开发的装备库模块拆分成可逐步迁移的任务，而不是一次性整块重写。

本轮目标不是“立即迁完整个装备库”，而是输出可执行拆分：

- 页面依赖列表
- 云开发残留点
- 建议后端接口清单
- 数据库迁移建议
- 前后端切换顺序

## 三、当前云开发残留点

### 3.1 列表页

[list.js](/Users/tommy/GearSage/GearSage-client/pkgGear/pages/list/list.js) 当前仍直接使用：

- `wx.cloud.database()`
- `db.collection(...).where(...).get()`

主要用途：

- 读取品牌表
- 读取装备主表
- 本地做搜索与筛选

### 3.2 详情页

[detail.js](/Users/tommy/GearSage/GearSage-client/pkgGear/pages/detail/detail.js) 当前仍直接使用：

主要用途：

- 读取装备主记录
- 读取型号变体明细

## 四、建议后端接口清单

建议在 `GearSage-api` 新增以下接口：

### 4.1 装备品牌

- `GET /mini/gear/brands?type=reels|rods|lures`

返回：

- `id`
- `name`

### 4.2 装备列表

- `GET /mini/gear/list`

建议 query：

- `type`
- `page`
- `pageSize`
- `keyword`
- `brand`
- `system`
- `action`
- `waterColumn`
- `options`
- `brakeSys`

### 4.3 装备详情

- `GET /mini/gear/detail?id=:id&type=:type`

建议返回：

- 主记录
- 变体列表
- 品牌信息

### 4.4 关联帖子

优先复用现有帖子接口，不单独新建：

- `GET /mini/topic/all`

建议继续支持以下筛选：

- `topicCategory`
- `gearCategory`
- `gearModel`
- `gearItemId`

## 五、数据库迁移建议

建议 GearSage-api 侧新增独立装备表，而不是继续依赖云数据库集合名。

推荐最小拆分如下：

- `gear_brands`
- `gear_reels`
- `gear_rods`
- `gear_lures`
- `gear_reel_variants`
- `gear_rod_variants`

第一阶段也可以折中：

- 表名暂保持接近现有语义，降低迁移成本
- 但必须迁到 PostgreSQL，不能继续让小程序直查 `wx.cloud.database()`

## 六、建议迁移顺序

### 阶段 1

- 后端先补品牌接口
- 后端补装备列表接口
- 前端列表页先从云数据库切到 HTTP

### 阶段 2

- 后端补装备详情接口
- 后端补变体明细返回
- 前端详情页从云数据库切到 HTTP

### 阶段 3

- 关联帖子改为复用独立后台帖子接口
- 清理详情页 `wx.cloud.callFunction('topic.all')`

### 阶段 4

- 清理 `gear.detail` 云函数依赖
- 清理导入脚本中的 `cloud://` 路径

## 七、当前结论

- 装备库是当前非 P0 最大残留块
- 这一轮不建议强行整块迁完
- 最合理的推进方式是：
  - 先出接口契约
  - 再切列表页
  - 再切详情页
  - 最后清导入与历史云资源
