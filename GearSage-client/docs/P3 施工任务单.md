## 二、P3 范围拍板

本阶段只包含以下 3 类事项：

1. 任务与商城深化
2. 装备库搜索 / 推荐 / 筛选继续产品化

本阶段明确不做：

## 三、P3 目标

### 3.1 任务与商城从“可用”走向“可持续扩展”

目标：

- 把当前任务与商城第一版继续做深
- 为后续任务规则、商品类型、积分策略扩展预留清晰结构

完成标准：

- 任务与商城不再只是最小可用闭环
- 关键业务口径、边界行为、后续扩展方向有明确拆分

### 3.2 装备库从“能查能筛”走向“更像产品能力”

目标：

- 继续深化搜索、推荐、筛选体验
- 把“数据覆盖不足”和“功能行为错误”继续分离清楚

完成标准：

- 搜索、推荐、筛选规则更稳定
- 用户能更清楚理解当前推荐词、家族覆盖和筛选结果
- 后续新装备补录时，搜索体验不需要频繁做临时修补

### 3.3 历史云兼容按批次退出

目标：

- 依据 `P2 cloud 兼容清理建议.md` 分批退出历史 `cloud://` 兼容
- 先收紧范围，再逐步撤除运行时兜底

完成标准：

- 运行时兼容与归档资产边界更清晰
- 不再让旧云开发逻辑继续扩散到新施工
- 为后续真正删除工具层兼容打好前置条件

## 四、P3 施工分组

## P3-A 任务与商城深化

### A1. 任务规则深化

重点：

- 任务分类与展示口径
- 重复领取、达成判定、奖励发放边界
- 后续任务规则扩展的拆分方向

验收标准：

- 任务行为不再只依赖“当前够用”
- 关键规则和边界有文档化口径

### A2. 商城能力深化

重点：

- 商品兑换口径
- 重复兑换、积分扣减、已拥有商品处理
- 后续虚拟权益以外商品扩展的接口与数据结构预留

验收标准：

- 商城行为更稳定、提示更一致
- 后续扩展不需要推翻当前第一版结构

---

## P3-B 装备库产品化深化

### B1. 搜索继续深化

重点：

- 关键词搜索命中规则
- `searchData/Data.js` 与 gear 主数据协同
- 结果可解释性

验收标准：

- 搜索结果更稳定
- 命中行为更可解释

### B2. 推荐继续深化

重点：

- 推荐词与推荐家族展示
- 覆盖态提示
- 数据缺失时的产品提示方式

验收标准：

- 推荐行为更符合用户预期
- 不再把“待补数据”误显示成“功能异常”

### B3. 筛选继续产品化

重点：

- 当前有效筛选项继续打磨
- 后续朝筛选元数据化推进
- 保持“展示出来就必须能用”

验收标准：

- 筛选行为更稳定
- 后续新增装备时筛选维护成本下降

---

## P3-C `cloud://` 历史兼容退出

### C1. 第一批：口径收紧

重点：

- 不再新增新的 `cloud://` 兼容扩散点
- 旧云脚本 / 云函数转为归档视角

验收标准：

- 主链路不再继续引入新的历史云依赖

### C2. 第二批：历史资产归零观察

重点：

- 观察是否仍有页面真实命中 `cloud://`
- 明确哪些兼容仍在被使用

验收标准：

- 为后续删除运行时兼容提供证据

### C3. 第三批：分层退出运行时兼容

重点：

- 页面层先退
- 工具层后退
- 最后处理 `tempUrlManager`

验收标准：

- 兼容逻辑按顺序退出
- 不因过早删除导致旧资产显示回归故障

---

## 五、P3 当前执行顺序建议

建议按以下顺序推进：

1. 先做任务与商城深化
2. 再做装备库搜索 / 推荐 / 筛选产品化
3. 最后按批次推进 `cloud://` 历史兼容退出

原因：

- 任务与商城是在现有第一版基础上最适合继续深化的业务块
- 装备库当前已经有较稳定底座，适合继续产品化
- `cloud://` 退出更适合建立在前两项稳定之后分批收口

## 六、P3 完成标准

P3 阶段完成时，应满足：

1. 任务与商城已从第一版可用继续深化到可持续扩展
2. 装备库搜索 / 推荐 / 筛选更接近产品化状态
3. `cloud://` 历史兼容已按批次推进退出，而不是继续长期悬挂
4. 外网联调、备案、海外迁移仍未混入本阶段施工
