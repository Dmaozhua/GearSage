# 发布页草稿保存与发布逻辑梳理

检查日期：2026-05-14

本文档只整理当前代码里的真实行为，用于后续自测和回归。它不是新的接口设计稿。

## 结论

发布页的“返回保存草稿”逻辑现在是发布页统一逻辑，不是某一个表单组件自己的逻辑。

当前可见发布入口包含：

- 求推荐
- 装备经验
- 提问

代码中仍保留以下历史/兼容模式，用于已有草稿或旧入口回填：

- 好物速报
- 鱼获展示
- 钓行分享

所有模式在进入具体编辑表单后，顶部返回和系统右滑返回都会走同一个退出检查：

1. 没有有效编辑内容，直接返回。
2. 内容没有相对初始快照发生变化，直接返回。
3. 有有效内容且发生变化，弹出“保存草稿”确认框。
4. 点击“保存”后保存草稿，再返回上一页。
5. 点击“不保存”后不调用保存接口，直接返回上一页。

## 入口与模式

### 可见发布入口

`PUBLISH_MODE_CONFIG` 当前只有三个可见模式。

| 入口 | modeKey | formModeKey | topicCategory | 关键区分字段 |
| --- | --- | --- | --- | --- |
| 求推荐 | `question_recommend` | `question` | `2` | `questionType = recommend` |
| 装备经验 | `experience` | `experience` | `1` | 无 |
| 提问 | `question_ask` | `question` | `2` | `questionType = ask` |

求推荐和提问共用 `post-question` 表单组件，后端也都属于 `topicCategory = 2`。二者靠 `questionType` 区分：

- `questionType = recommend`：回填为求推荐
- `questionType = ask`：回填为提问
- 草稿里缺少 `questionType` 时，兼容回填为求推荐

### 历史/兼容模式

`LEGACY_MODE_CONFIG` 仍保留以下模式：

| 模式 | modeKey | formModeKey | topicCategory |
| --- | --- | --- | --- |
| 好物速报 | `recommend` | `recommend` | `0` |
| 装备经验 | `experience` | `experience` | `1` |
| 讨论&提问 | `question` | `question` | `2` |
| 鱼获展示 | `catch` | `catch` | `3` |
| 钓行分享 | `trip` | `trip` | `4` |

这些模式主要用于旧数据、旧草稿、旧入口的兼容回填。当前发布模式选择页不会主动展示好物速报、鱼获展示、钓行分享。

## 退出检查

发布页 WXML 在进入具体编辑页后挂载 `page-container`：

- `showModeSelection = true` 时，仍在选择发布模式，不启用草稿退出拦截。
- `showModeSelection = false` 时，启用 `page-container` 监听系统右滑/原生返回。

两个返回入口都会进入 `confirmDraftBeforeExit()`：

- 页面内顶部“返回”：`handlePageExit()`
- 系统右滑/原生返回：`onNativeBeforeLeave()`

退出提示出现的条件由 `shouldEnableExitGuard()` 决定：

- 当前不是允许离开的内部状态
- 当前没有在加载草稿
- 当前已经进入具体编辑表单
- 当前存在发布模式
- 当前数据相对初始快照发生变化
- 当前存在有意义的草稿内容

只要以上条件不同时满足，就直接返回上一页，不弹提示。

## 初始快照与内容判断

发布页会在以下时机记录初始快照：

- 用户选择某个发布模式后
- 从求推荐相关入口带预填内容进入后
- 从“我的-我的发布”进入草稿编辑并完成回填后
- 成功保存草稿后

快照使用 `getCurrentPostData()` 生成当前帖子 payload，再通过 `sanitizeComparableValue()` 做稳定化后 JSON 序列化。`rejectReason` 不参与变化判断。

这意味着：

- 刚进入新建表单，还没有编辑时，不会提示保存草稿。
- 刚进入草稿编辑，还没有改动时，不会提示保存草稿。
- 成功保存草稿后，当前内容会成为新的基准，再次返回不会重复提示。
- 只改了拒审原因这类展示字段，不会触发保存草稿。

“有意义内容”由 `hasMeaningfulDraftContent()` 判断：

- 非空字符串算有效内容
- 大于 0 的数字算有效内容
- `true` 布尔值算有效内容
- 数组/对象里只要有任一有效值就算有效内容
- `id`、`topicCategory`、`rejectReason` 不算用户编辑内容

## 点击“不保存”

当前“不保存”分支行为：

1. 关闭保存草稿弹窗状态。
2. 设置允许离开标记。
3. 关闭 `page-container` 的返回拦截。
4. 延迟执行 `wx.navigateBack()` 返回上一页。
5. 不调用 `api.createTopic()`。
6. 不改动已有草稿。

如果当前是新建帖子，“不保存”会丢弃本次编辑内容。

如果当前是编辑已有草稿，“不保存”会丢弃本次修改，但服务器上原来的草稿仍保持上一次保存的内容。

## 点击“保存”

当前“保存”分支行为：

1. 展示 `保存草稿中...` loading。
2. 调用 `saveCurrentDraft()`。
3. `saveCurrentDraft()` 通过 `getCurrentPostData()` 组装当前模式 payload。
4. 调用 `api.createTopic(currentPayload)`。
5. `api.createTopic()` 实际请求为 `PUT /mini/topic`。
6. 保存成功后，更新 `editingDraftId` 和 `formData.id`。
7. 重新捕获初始快照。
8. 隐藏 loading。
9. 展示 tip：`草稿放在 我的-我的发布`。
10. 延迟返回上一页。

保存失败时：

- 隐藏 loading
- 展示错误提示
- 停留在当前页面
- 重新恢复退出拦截

## 同一个草稿覆盖规则

同一个帖子只能保存成一个草稿，关键取决于 payload 里的 `id`。

### 新建帖子首次保存

新建时没有 `id`：

1. 前端调用 `PUT /mini/topic`，payload 不带有效 `id`。
2. 后端执行 insert。
3. 新建一条 `status = 0` 的草稿。
4. 后端返回新草稿 `id`。
5. 前端把这个 `id` 写回 `editingDraftId` 和 `formData.id`。

### 再次编辑同一个草稿

从“我的-我的发布”进入草稿时：

1. 页面 URL 带 `draftId`。
2. 前端通过 `getTopicDetail(draftId)` 拉取草稿详情。
3. 回填表单时把 `topic.id` 写入 `formData.id`。
4. 页面状态里也保存 `editingDraftId = topic.id`。

再次点击“保存”时：

1. `getCurrentPostData()` 会把 `formData.id || editingDraftId` 带入 payload。
2. `PUT /mini/topic` payload 带原草稿 `id`。
3. 后端 `saveTopicDraft()` 命中 `dto.id` 分支。
4. 后端只更新同一个用户、同一个 `id` 的原帖子。
5. 不会创建第二条草稿。

因此，用户先创建“提问帖子 a”并保存草稿，再进入草稿 a 编辑并保存，只会覆盖草稿 a，不会生成“提问帖子 b”草稿。

## 发布逻辑

发布按钮不会直接调用发布接口，而是先进入预览页。

发布页行为：

1. 表单组件触发 submit。
2. 发布页调用 `openPreview(type, formData)`。
3. `openPreview()` 调用 `buildPostData(type, formData)`。
4. 将 `{ postData, selectedMode, previewTime }` 写入本地缓存 `topic_preview_payload_v1`。
5. 跳转到 `/pkgContent/postPreview/postPreview`。

预览页行为：

1. 从本地缓存读取 `postData`。
2. 点击确认发布后调用 `api.publishTopic(postData)`。
3. `api.publishTopic()` 实际请求为 `POST /mini/topic`。
4. 发布成功后移除预览缓存。
5. 展示 `已提交审核`。
6. `wx.navigateBack({ delta: 2 })` 返回发布入口之前的页面。

后端发布行为：

- 如果 payload 带 `id`，发布同一个草稿并更新状态。
- 如果 payload 不带 `id`，发布一条新帖子。
- 发布会走文本审核。
- 审核结果为人工审核或强制审核时，状态为 `1`。
- 审核通过且无需人工审核时，状态为 `2`。
- 发布成功会清空 `rejectReason`。

## 前端到后端的数据转换

小程序调用 `createTopic()` 和 `publishTopic()` 前，都会经过 `buildTopicPayload()`。

转换重点：

- `mainContent` 会被转换为后端 `content`。
- `mainImages` 会被转换为后端 `images`。
- `gearCategory`、`questionType`、`recommendMeta` 等结构化字段会进入 `extra`。
- 已知顶层业务字段会被归入 `extra`，后端详情再 normalize 回小程序可读字段。

这个转换对草稿保存很关键：

- 如果表单只给 `mainContent/mainImages`，后端仍能收到 `content/images`。
- 如果装备经验选择了“渔轮”，`gearCategory` 会随 `extra` 保存，并在草稿详情 normalize 后回填。

## 后端草稿接口行为

`PUT /mini/topic` 对应 `TopicService.saveTopicDraft()`。

### payload 带 id

后端执行 update：

- 更新同一用户、同一 topic id 的帖子
- 设置 `status = 0`
- 设置 `isDelete = 0`
- 更新时间
- 删除该帖相关的 `topic_rejected` 消息
- 不创建新草稿

如果找不到同一用户的该 id，返回 404。

### payload 不带 id

后端执行 insert：

- 新建一条 `status = 0` 的草稿
- `publishTime = NULL`
- `likeCount = 0`
- `commentCount = 0`
- `isDelete = 0`
- 返回新 id

## 当前仍需真机确认的点

代码层面已经统一了保存/不保存分支，但 `page-container` 的原生返回拦截属于微信运行时行为，最终仍需要微信开发者工具或真机确认：

- 点击“不保存”是否一次返回成功。
- 点击“保存”出现 tip 后是否自动返回。
- 右滑返回与顶部返回是否一致。
- loading、toast、navigateBack 的时序是否在低端机上稳定。

如果上述真机测试都通过，可以认为发布页不同模式的保存/不保存逻辑闭环完成。
