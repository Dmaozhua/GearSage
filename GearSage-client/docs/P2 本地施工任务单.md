## 三、P2 目标

### 3.1 本地可持续联调

目标：

- 当前已切后台页面在本地环境下可重复回归
- 每次本地回归有固定 smoke test 路径
- 出问题时能快速定位到页面、接口、数据层

完成标准：

- 已形成固定的本地 smoke test 清单
- 主要页面在微信开发者工具中可重复通过
- 关键接口状态码、回包字段、登录态切换表现稳定

### 3.2 已完成链路从“能跑”提升到“稳定”

目标：

- 把评论点赞、上传、标签、商城、任务从“第一版可用”继续收紧
- 优先处理重复点击、刷新后状态、边界回包、业务提示一致性

完成标准：

- 主要边界场景有明确处理口径
- 前后端回包与页面表现一致
- 不再把已知稳定性问题混在 P0 / P1 文档里继续滚动

### 3.3 装备库继续深化

目标：

- 继续打磨搜索、推荐、筛选体验
- 让当前 `rate/excel -> PostgreSQL -> searchData/Data.js` 链路在本地维护中更顺手

完成标准：

- 搜索、推荐、筛选行为有更清晰的规则说明
- 能区分“数据缺失导致无结果”和“功能逻辑错误”
- 装备库页面联调路径稳定

### 3.4 逐步退出历史云兼容

目标：

- 把仍保留的 `cloud://` 兼容分成“必须保留 / 可清理 / 待资产迁移后清理”
- 避免主链路继续被历史兼容逻辑拖复杂

完成标准：

- 输出一份后续清理建议
- 不在 P2 中盲目一次性删除全部兼容逻辑

## P2-B 稳定性收口

### B1. 评论点赞稳定性

重点：

- 主评论 / 回复点赞一致性
- 刷新后状态是否与当前用户一致
- 重复点击、登录切换、列表重拉后的表现

验收标准：

- 点赞状态与当前手机号账号一致
- 不因前端缓存导致跨账号串状态
- 已知问题和真正后端待收项有明确边界

当前进度：

- 已完成第一轮专项回归
- 已验证：
  - 评论新增后列表可回读新评论
  - 评论点赞二次切换返回稳定
  - 二次切换后 `isLiked / likeCount` 回读一致
- 当前备注：
  - 评论新增接口当前返回值仍为 `true`，前端依赖“发完后重拉列表”口径工作，不阻塞 P2-B

### B2. 上传稳定性

重点：

- token 过期后上传重试
- 草稿二次编辑时不重复上传已上传图片
- 头像 / 背景 / 发帖图片口径一致

验收标准：

- 上传失败提示可读
- 上传 URL 与页面回显一致
- 本地调试时不混入错误的 `cloud://` 写入

当前进度：

- 已完成第一轮回归
- 已验证：
  - `image / avatar / background` 三类上传可稳定返回本地 URL
  - 上传后的资料更新、草稿保存、帖子发布链路均可复用上传结果
  - 本地回归中未出现新的 `cloud://` 写入

### B3. 标签 / 商城稳定性

重点：

- 新账号默认积分与 starter tags
- 标签兑换、佩戴、重复兑换提示
- 商城扣积分后列表和余额刷新

验收标准：

- `my-tags / tag-manager / points-exchange` 一致
- 业务错误走用户可读提示

当前进度：

- 已完成第一轮专项回归
- 已验证：
  - 新账号默认 `10000` 积分与 starter tags 生效
  - 商城首次兑换成功
  - 重复兑换返回 `400` 且业务文案为“已拥有该标签”
  - 重复兑换不会二次扣积分
  - 标签非法佩戴返回明确业务错误
  - 标签合法佩戴保存稳定

### B4. 任务稳定性

重点：

- 任务展示、领取、积分发放
- 每日任务 / 更多任务 / 成就分类结果
- 重复领取与边界提示

验收标准：

- 领取后积分刷新稳定
- 页面分类和接口回包口径一致

当前进度：

- 已完成第一轮专项回归
- 已验证：
  - 任务列表、未完成列表、领取接口可正常联动
  - 首次领取后积分正常增加
  - 重复领取返回稳定

## P2-C 装备库深化

### C1. 搜索逻辑深化

重点：

- 关键词搜索
- `searchData/Data.js` 命中行为
- 本地搜索辅助数据与 PostgreSQL 装备主数据的一致性

验收标准：

- 能明确区分“命中不到数据”与“搜索逻辑错误”
- 搜索行为可解释、可复现

### C2. 推荐词 / 推荐家族深化

重点：

- 推荐词展示
- 点击推荐词后的严格筛选行为
- 推荐家族与当前数据覆盖范围的说明

验收标准：

- 点某个推荐词时，不再混入其他推荐家族结果
- 当前数据缺失时允许返回空，但不应错误回退成混合结果

### C3. 筛选深化

重点：

- 品牌、类别、系统、水层、动作等当前有效筛选
- 继续识别“前端有配置但当前数据不支持”的筛选项
- 后续可演进为元数据化，但本阶段先以行为正确为优先

验收标准：

- 当前展示出来的筛选项必须真正生效
- 不再出现“显示可选但实际永远筛不出来”的假筛选

## 五、P2 当前执行顺序建议

建议按以下顺序推进：

1. 先整理本地 smoke test 清单
2. 处理评论点赞 / 上传 / 标签 / 商城 / 任务稳定性收口
3. 继续深化装备库搜索、推荐、筛选
4. 最后输出 `cloud://` 历史兼容逻辑后续清理建议

原因：

- 先把本地回归路径固定，后续每轮修改都能快速复测
- 先稳主链路，再深化装备库体验
- 把历史兼容建议放在末尾，更适合基于现状做分批清理
