# 消息系统 v0 方案

版本：v1.0  
状态：待实施  
适用对象：Tommy / Codex / 后端 / 前端 / 审核与通知链路

## 一、目标

这份文档用于定义 GearSage 消息系统 v0 的最小落地方案。

当前目标不是做完整 IM、私信或推送系统，而是：

> 先形成“审核结果有通知、互动行为有回执”的最小通知中心。

## 二、为什么现在要做

当前 GearSage 已经具备：

1. 内容审核第一轮
2. 审核后台 v0
3. 帖子通过 / 驳回 / 下架 / 恢复显示
4. 评论与点赞基础互动

如果缺少消息系统，用户会遇到两个明显问题：

1. 自己发的内容发生了什么，不知道
2. 被点赞、被评论了，也不知道

因此消息系统 v0 的价值不是锦上添花，而是：
- 给审核闭环一个用户侧出口
- 给社区互动一个最基本的反馈机制
- 给微信审核说明增加一层“用户治理与告知能力”

## 三、拍板结论

### 3.1 第一版只做通知，不做聊天

消息系统 v0 只做：
- 审核结果通知
- 互动行为通知

第一版明确不做：
- 私信
- 聊天
- push 推送
- 复杂订阅设置
- 站外通知通道
- 多端已读同步复杂设计

### 3.2 第一版消息类型

第一版只做以下 6 类：
1. `topic_approved`
2. `topic_rejected`
3. `topic_removed`
4. `topic_restored`
5. `comment_received`
6. `like_received`

### 3.3 第一版用户收益

用户能明确知道：
- 帖子审核通过了
- 帖子被驳回了
- 帖子被下架了
- 帖子恢复显示了
- 有人评论了自己
- 有人点赞了自己

## 四、与当前项目状态的对齐

当前 GearSage 已经具备：
- 帖子状态体系：`0 草稿 / 1 待审核 / 2 已发布`
- 审核后台 v0：可对帖子执行通过 / 驳回 / 下架 / 恢复显示
- 评论与点赞链路：已可读写、已接入小程序页面
- 审核日志与规则页：已可用

因此消息系统 v0 不需要推翻现有设计，而是：

> 在现有业务动作发生时，补一张“用户通知表”和一层通知写入逻辑。

## 五、最小数据结构设计

建议新增表：`user_messages`

最低字段：
- `id`
- `userId`
- `type`
- `title`
- `content`
- `targetType`
- `targetId`
- `isRead`
- `extra`
- `createTime`

### 字段说明

`userId`
- 消息接收人

`type`
- 消息类型，第一版枚举：
  - `topic_approved`
  - `topic_rejected`
  - `topic_removed`
  - `topic_restored`
  - `comment_received`
  - `like_received`

`title`
- 消息标题，便于前端直接渲染

`content`
- 消息正文，给用户看的可读说明

`targetType`
- 目标类型：`topic` / `comment`

`targetId`
- 目标 ID

`isRead`
- 是否已读，默认 `0`

`extra`
- JSON，用于补充：
  - 审核原因
  - 帖子标题
  - 评论摘要
  - 操作者类型
  - 跳转所需扩展字段

`createTime`
- 创建时间

## 六、最小接口设计

第一版建议接口：

### 6.1 `GET /mini/message`
用途：
- 拉取当前用户消息列表

建议 query：
- `page`
- `limit`
- `type`（可选）
- `onlyUnread`（可选）

返回：
```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [],
    "unreadCount": 0
  }
}
```

### 6.2 `POST /mini/message/read`
用途：
- 标记单条消息已读

请求参数：
- `messageId`

返回：
```json
{
  "code": 0,
  "message": "ok",
  "data": true
}
```

### 6.3 `POST /mini/message/read-all`
用途：
- 全部标记已读

返回：
```json
{
  "code": 0,
  "message": "ok",
  "data": true
}
```

第一版不做：
- 删除消息
- 消息设置
- 推送订阅接口

## 七、后端落点建议

与当前 NestJS 结构保持一致：

```text
src/modules/message/
  message.controller.ts
  message.service.ts
  dto/
```

并在 `app.module.ts` 中继续直接注册 controller / provider。

`MessageService` 负责：
1. 写消息
2. 查消息
3. 标记已读
4. 统计未读数

`MessageController` 负责：
- `GET /mini/message`
- `POST /mini/message/read`
- `POST /mini/message/read-all`

## 八、消息写入接入点

### 8.1 帖子审核通过
接入点：
- `admin-review.service.ts`

动作：
- 给帖子作者写 `topic_approved`

建议文案：
- 标题：`你的帖子已通过审核`
- 正文：`你发布的《xxx》已通过审核，现在其他用户可以看到了。`

### 8.2 帖子驳回
接入点：
- `admin-review.service.ts`

动作：
- 给帖子作者写 `topic_rejected`

建议文案：
- 标题：`你的帖子未通过审核`
- 正文：`你发布的《xxx》未通过审核，请修改后重新发布。`

如果后台已有备注 / 驳回理由：
- 通过 `extra.reason` 传给前端显示

### 8.3 帖子下架
接入点：
- `admin-review.service.ts`

动作：
- 写 `topic_removed`

建议文案：
- 标题：`你的帖子已被下架`
- 正文：`你发布的《xxx》当前已被下架。`

### 8.4 帖子恢复显示
接入点：
- `admin-review.service.ts`

动作：
- 写 `topic_restored`

建议文案：
- 标题：`你的帖子已恢复显示`
- 正文：`你发布的《xxx》已恢复正常展示。`

### 8.5 被评论
接入点：
- `comment.service.ts`

动作：
- 在评论成功写入后，给帖子作者写 `comment_received`
- 第一版先只给帖子作者发，不先做回复评论通知

建议文案：
- 标题：`有人评论了你`
- 正文：`你的帖子《xxx》收到了新评论。`

### 8.6 被点赞
接入点：
- `topic.service.ts`

第一版建议先做：
- 帖子被点赞通知

建议文案：
- 标题：`有人点赞了你`
- 正文：`你的帖子《xxx》收到了一个赞。`

## 九、页面最小落地方案

第一版只需要一个：

### 9.1 消息中心页
建议路径：
- `pages/message/index`

展示形式：
- 按时间倒序列表
- 每条展示：
  - 标题
  - 正文摘要
  - 时间
  - 未读态

点击行为：
- 标记已读
- 能跳到对应帖子详情 / 对应内容页

### 9.2 首页 / 我的页入口
第一版建议：
- 先在“我的”页加一个消息入口
- 可以附带未读数

第一版不做：
- 独立 tab
- 按类型复杂分组
- 高级筛选
- 站内弹窗中心

## 十、前端口径建议

审核通知文案：
- 审核通过：`你的帖子已通过审核`
- 驳回：`你的帖子未通过审核，请修改后重新发布`
- 下架：`你的帖子已被下架`
- 恢复：`你的帖子已恢复显示`

互动通知文案：
- 评论：`有人评论了你`
- 点赞：`有人点赞了你`

红点策略：
- 第一版只做未读数
- 进入消息页后支持单条 / 全部已读

## 十一、与审核后台 v0 的关系

消息系统 v0 不替代审核后台。

关系是：
- 审核后台负责治理动作
- 消息系统负责把治理结果告诉用户

这两者一起构成完整的审核闭环。

## 十二、实施顺序建议

1. 新增 `user_messages`
2. 落后端 `message.service.ts`
3. 打通 4 个审核类通知：
   - `topic_approved`
   - `topic_rejected`
   - `topic_removed`
   - `topic_restored`
4. 再补互动类通知：
   - `comment_received`
   - `like_received`
5. 最后补消息中心页

## 十三、第一版完成标准

满足以下条件即可视为 v0 完成：
1. 审核通过 / 驳回 / 下架 / 恢复会产生通知
2. 评论 / 点赞至少有一类互动通知可用
3. 小程序内有消息中心页
4. 用户能标记已读
5. 前后端不引入复杂新框架
6. 不影响当前 `{ code, message, data }` 主口径

## 十四、一句话结论

消息系统 v0 不是为了做“社区热闹”，而是为了：

> 让审核结果和互动行为对用户有回执，补齐 GearSage 当前最小可信社区闭环。
