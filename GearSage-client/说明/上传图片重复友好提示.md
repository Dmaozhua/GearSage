# 要点总结

1. 在组件内部维护每张图片的“特征值（signature）”，key 用云 `fileID`，value 为 signature。
2. 在用户 `wx.chooseImage` 后、上传前计算新图片 signature，与已有 signatures 比较；若检测到相似项，**提示**“有相似图片，建议优化”，但**不阻止上传**。
3. 上传完成后把 signature 映射到返回的 `fileID`（用于后续删除判别）。
4. 删除图片时同时删除对应的 signature。
5. signature 采用轻量且稳健的构造（尺寸 + 宽高比 + 文件大小区间），可抵抗微信上传时的格式转换/压缩带来的差异。
6. 所有变更不改动父组件事件接口（`chooseimage` / `deleteimage`）和最终提交格式（`formData.mainImages` → 逗号分隔 `contentImages`），仅组件内部维护 signature（不随表单提交）。

# 新增工具：`utils/imageSignature.js`

```javascript
// utils/imageSignature.js
// 轻量级图片特征值计算与相似度检测
// 思路：使用宽高、宽高比、文件大小区间作为 signature，能在格式转换/压缩后仍保持较高识别率

const calculateImageSignature = (tempFilePath, fileSize = 0) => {
  // tempFilePath: 本地临时路径（wx.chooseImage 返回）
  // fileSize: 可选（来自 chooseResult.tempFiles[i].size），单位 bytes，若可用请传入以提高稳定性
  return new Promise((resolve, reject) => {
    wx.getImageInfo({
      src: tempFilePath,
      success: (res) => {
        const width = res.width || 0;
        const height = res.height || 0;
        const aspect = (height === 0) ? 0 : parseFloat((width / height).toFixed(3));
        // 将文件大小归到区间，比如按 10KB 为单位分桶（避免上传时压缩带来的微小变化）
        const sizeBucket = fileSize ? Math.round(fileSize / 10240) : 0;
        const signature = `${width}x${height}_a${aspect}_s${sizeBucket}`;
        resolve(signature);
      },
      fail: (err) => {
        // 若获取信息失败，返回一个最保守的 signature（空或基于文件大小）
        if (fileSize) {
          const sizeBucket = Math.round(fileSize / 10240);
          resolve(`unknown_unknown_a0_s${sizeBucket}`);
        } else {
          reject(err);
        }
      }
    });
  });
};

// 判断两个 signature 是否“相似”
const isSignatureSimilar = (sig1, sig2) => {
  if (!sig1 || !sig2) return false;
  if (sig1 === sig2) return true;

  // 解析 signature: widthxheight_a<aspect>_s<sizeBucket>
  const parse = (s) => {
    const [sizePart, aPart, sPart] = s.split('_');
    const [w, h] = (sizePart && sizePart.includes('x')) ? sizePart.split('x').map(Number) : [0,0];
    const aspect = aPart ? parseFloat(aPart.replace('a','')) : 0;
    const sizeBucket = sPart ? parseInt(sPart.replace('s',''), 10) : 0;
    return { w, h, aspect, sizeBucket };
  };

  const p1 = parse(sig1);
  const p2 = parse(sig2);

  // 规则：
  // 1) 宽高完全相同 -> 相似
  if (p1.w && p2.w && p1.w === p2.w && p1.h === p2.h) return true;

  // 2) 宽高比接近（容差 0.02）且 sizeBucket 相同 -> 相似
  if (Math.abs(p1.aspect - p2.aspect) < 0.02 && p1.sizeBucket === p2.sizeBucket) return true;

  // 3) 宽高比接近且宽或高其中之一相同 -> 相似
  if (Math.abs(p1.aspect - p2.aspect) < 0.02 && (p1.w === p2.w || p1.h === p2.h)) return true;

  return false;
};

const checkSimilarImages = (newSignature, existingSignatures) => {
  // existingSignatures: { fileID: signature, ... }
  if (!newSignature) return false;
  const vals = Object.values(existingSignatures || {});
  for (const ex of vals) {
    if (isSignatureSimilar(newSignature, ex)) return true;
  }
  return false;
};

module.exports = {
  calculateImageSignature,
  checkSimilarImages,
  isSignatureSimilar
};
```

# 设计/实现说明与注意事项（必读）

1. **为什么不直接用哈希（md5）？**

   * 微信小程序上传/压缩可能改变像素或格式（PNG -> JPEG），直接对上传后文件做精确哈希可能出现不同值。这里用“尺寸 + 宽高比 + 文件大小区间”的组合可以在绝大多数场景下将相同图片（即便被压缩/转码）识别为相似。若需要更高精度，可考虑在服务器端解码并计算感知哈希（pHash）或在小程序里绘制到 canvas 采样像素并计算，但性能与支持会受限。

2. **签名存在哪里？**

   * 签名只存在组件内部（`this.data.imageSignatures`），不会随 `formData` 提交到后端，满足原始需求（只是提示）。如果你希望服务器端也做去重校验，建议在 `buildPostData` 时也生成并提交轻量 signature 列表，或在后端基于 fileID 做相似度检查。

3. **上传工具兼容性**：

   * 如果现有 `imageUploadUtils.chooseAndUploadImages` 把选择和上传都封装在一起，我们无法在上传前拿到 `tempFilePath` 去计算 signature。推荐把工具拆成 `choose` 与 `upload` 两步（即增加 `uploadImages`），这样可以在上传前做检查并友好提示。

4. **性能**：

   * 计算 signature 主要调用 `wx.getImageInfo`，这是轻量操作。我们只在用户选择图片时执行一次，不会影响 UI 流畅性。若担心大量图片同时选择（比如 9 张），可以考虑并发限制（例如同时并发 3 个 getImageInfo），但通常不必要。

5. **容错**：

   * 若 signature 计算失败（极少数情况下），会 **不影响上传流程**，只记录日志并继续上传（这是你要求的“友好提示，不阻拦用户上传”的核心）。

6. **提示文案**：

   * 我使用了你给出的提示文案：`有相似图片，建议优化`（toast，3s）。如需更醒目也可以改成 modal，但 modal 会打断流畅度，建议保持 toast。
