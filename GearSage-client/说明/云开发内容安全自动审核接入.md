# 微信小程序 · 云开发内容安全自动审核接入
- 版本：v1.0  
- 日期：2026-02-27  
- 目标：所有**文本输入点**与**图片上传点**在提交/绑定前进行自动审核；不通过则**阻断**并提示。

---

## 一、需求文档（PRD）

### 1. 背景
小程序存在 UGC（用户生成内容）：昵称、个人描述、发帖、评论、头像/背景/帖子图片等。需要对涉黄、涉政、违法违规内容进行拦截，降低合规风险，同时保证业务体验可控。

### 2. 目标
1. **文本审核**：所有文本输入点（用户名、个人描述、发帖、评论等）在“保存/发布/提交”前必须通过文本安全检测；未通过则**阻断提交并提示**。  
2. **图片审核**：所有图片上传点（头像、背景图、发帖图片等）在“最终绑定到业务数据”前必须通过图片安全检测；未通过则**阻断并提示**，并确保违规图片不落库、不对外可见。  
3. 提供统一的“内容安全网关”能力，前端接入成本低、后端可审计可追踪。

### 3. 范围

#### 3.1 文本审核覆盖点（必须）
- 用户资料
  - 用户名/昵称
  - 个人描述/签名
- 发帖
  - 帖子标题
  - 帖子正文
- 评论
  - 评论内容
- 其他所有会对外展示的用户输入字段（话题名、标签、装备描述等）

#### 3.2 图片审核覆盖点（必须）
- 用户资料
  - 头像
  - 背景图
- 发帖
  - 帖子图片（单图/多图）
- 其他（如存在）
  - 评论图片、举报截图等

#### 3.3 不在本期范围（可扩展）
- 视频/音频审核
- “发帖规则不符合”的人工审核后台流程（另起 PRD）

### 4. 关键原则
- **前端校验只用于体验优化**，真正拦截必须在**云函数（服务端）**完成：所有写库/更新资料/发布内容接口必须二次审核，防抓包绕过。
- 审核失败：**不写库、不绑定图片、不发布**；提示用户修改后再提交。
- 云函数需在 `config.json` 声明 openapi 权限，否则会报 `-604101`；更新权限后存在缓存时间（约 10 分钟）。  
  参考：CloudBase FAQ（-604101 权限错误）  
  https://docs.cloudbase.net/faq/knowledge/cloud-call-604101-permission-error

### 5. 业务流程

#### 5.1 文本流程（同步拦截）
1. 用户在任意文本输入点点击【保存/发布/提交】  
2. 前端调用云函数 `sec_text_check`（传入 `scene` + `content`）  
3. 云函数调用 `security.msgSecCheck` 校验  
4. 结果：
   - 通过：继续业务提交（写库/更新资料）
   - 不通过：**阻断**，前端弹出提示

> 实践中常把 `errCode === 87014` 视为“内容违规”。  
> 参考：腾讯云社区文章（msgSecCheck/87014）  
> https://cloud.tencent.com/developer/article/1733693

#### 5.2 图片流程（阻断最终绑定）
图片审核需要“两阶段提交”（避免违规图片进入业务数据）：

**阶段 A：临时上传（不入库、不对外）**  
1. 用户选择图片  
2. 前端上传到云存储临时路径（如 `tmp/`）  
3. 前端立刻调用云函数 `sec_image_check`，传 `fileID + scene`

**阶段 B：审核通过才正式绑定**  
4. 通过：把 `fileID` 写入用户资料/帖子数据，或复制/移动到正式路径  
5. 不通过：删除该 `fileID`（云函数删或前端兜底删），提示用户；业务数据不落库

> `mediaCheckAsync` 是异步接口：成功返回 `trace_id`，结果可能延迟回调。若要求“立刻阻断”，建议本期采用同步 `imgSecCheck`，并可加异步复检兜底。  
> 参考：CloudBase 社区讨论（mediaCheckAsync 异步特性）  
> https://cnb.cool/tencent/cloud/cloudbase/community/-/issues/380

### 6. 数据要求

#### 6.1 审核结果记录（必须）
建议记录所有审核行为用于追踪排查（不保存原文/原图，只存 hash/引用）：

集合：`sec_audit_log`
- `_id`
- `uid/openid`
- `scene`（如 `PROFILE_NICKNAME` / `POST_BODY` / `AVATAR`）
- `contentHash`（文本 hash 或 fileID）
- `result`：`PASS | REJECT | ERROR`
- `providerErrCode`（如 87014 / -604101）
- `createdAt`

#### 6.2 业务集合字段（建议）
- `users`：`nickName`、`bio`、`avatarFileId`、`bgFileId`
- `posts`：`title`、`content`、`images[]`、`createdAt`、`updatedAt`

### 7. 提示文案（统一）
- 文本不通过：`内容包含违规或敏感信息，请修改后再提交。`
- 图片不通过：`图片未通过安全审核，请更换后重新上传。`
- 服务异常：`审核服务繁忙，请稍后重试。`

> 不建议提示具体原因（涉黄/涉政等），避免对抗性绕过。

### 8. 非功能需求
- 性能：不在每次输入字符时调用；仅在 **blur（可选）+ 提交（必须）**触发。
- 稳定性：审核异常默认 **fail-closed（阻断）**。
- 安全：后台代编辑/代发也应复用同一套审核策略。

### 9. 验收标准
- 任意文本输入点：提交前必调用 `sec_text_check`，不通过则不写库并提示。
- 任意图片上传点：最终绑定前必审核；不通过则删除临时文件并提示，业务集合中不得出现该 `fileID`。
- 配置缺失权限时报 `-604101`，按文档配置后可恢复。  
  https://docs.cloudbase.net/faq/knowledge/cloud-call-604101-permission-error
- `sec_audit_log` 可查到每次审核记录。

## 二、技术文档（技术方案）

### 1. 总体架构
- 小程序前端：统一调用 `ContentGuard` 工具封装
- 云函数层：
  - `sec_text_check`（同步文本审核）
  - `sec_image_check`（同步图片审核）
  - 可选：`sec_media_submit`/`sec_media_callback`（异步兜底）
- 云数据库：业务集合 + `sec_audit_log`
- 云存储：`tmp/` 临时目录 + `public/` 正式目录

### 2. 云函数权限配置（必须）
在云函数目录下创建/修改 `config.json`：

```json
{
  "permissions": {
    "openapi": [
      "security.msgSecCheck",
      "security.imgSecCheck",
      "security.mediaCheckAsync"
    ]
  }
}
