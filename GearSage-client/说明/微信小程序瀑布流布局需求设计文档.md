微信小程序瀑布流布局需求设计文档
1. 功能概述
为微信小程序首页帖子列表增加瀑布流显示模式，在现有筛选栏目中新增布局切换按钮，允许用户在默认排列和瀑布流布局之间切换。

2. 功能需求
2.1 核心功能
新增布局切换按钮，支持"默认排列"和"瀑布流"两种显示模式

瀑布流布局实现两列交错的卡片排列

保持现有筛选和排序功能不变

支持两种布局模式下的图片懒加载和性能优化

2.2 用户交互
点击布局切换按钮可即时切换显示模式

切换布局时保持当前筛选状态和数据位置

瀑布流模式下支持无限滚动加载
3. 界面与交互设计
3.1 布局切换入口设计

位置：

位于首页帖子列表顶部筛选栏右侧，与排序筛选同一水平行。

在不挤占筛选空间的前提下优先使用图标 + 弱文案形式。

样式规范：

切换按钮形态：

使用双状态按钮：

默认排列：列表图标（横向三条 / 四条线）

瀑布流：两列不等高矩形图标

默认状态高亮当前布局，对应图标 + 下方 8px 字号文案（"列表" / "瀑布"）可选。

高亮态：主色/主题色 + 实心图标

非高亮态：中性/灰色线框图标

交互反馈：

点击立即切换布局，无二次确认。

切换时给 150–250ms 的轻微过渡动画（淡入、位移动画，避免突兀闪烁）。

按钮有点击态（降低透明度或加深颜色）。

状态记忆：

需在同一次会话内记住用户选择的布局模式。

建议增加本地缓存（storage）记忆用户上次选择，下次进入首页沿用。

4. 帖子卡片视觉与信息层级设计
4.1 默认排列模式（保持现有，增加规范说明）

保持你现在的「普通列表」逻辑，只做规范化，方便后续对比和实现。

信息结构（建议顺序）：

用户头像 + 昵称 + 等级/标识

发布时间 / 距今时间

标题（主信息）

内容摘要（1–2 行）

图片区域：最多展示 3 张缩略图（小图并排或 1 大 2 小）

Tag / 分类 / 城市 / 鱼种等标签

底部操作区：点赞数 / 评论数 / 收藏数 / 浏览量（精简为 2–3 个）

样式特点：

卡片高度相对统一，信息结构规整、便于扫读。

图片为内容补充，而不是背景。

更适合精准浏览、按时间/条件筛选场景。

4.2 瀑布流模式卡片设计（重点）

核心原则：
与默认列表样式拉开明显区分；更“视觉流”，更轻信息，适合刷图、灵感、种草体验。

整体结构：

两列卡片：列宽相同，卡片高度随图片比例和文案多少自适应。

每个卡片仅展示「一张图」作为主视觉背景（或顶部主图）。

用户与文本信息集中在卡片底部一块区域，信息量精简。

卡片内容元素（建议）：

背景图（必选）

来源：帖子首图 / 优先图。

占据卡片 70–85% 高度。

默认裁切为「上部完整，中下部可被渐变遮罩」。

圆角：8–12rpx（和全局风格统一）。

底部信息区（悬浮/覆盖在图片底部）：

使用下方 20–35% 区域的深色渐变遮罩（从透明到半透明黑或主色），保证文字对比度。

内容从左到右布局：

左侧：用户头像（24–28rpx）+ 用户昵称（单行，超出省略号）。

中部（或昵称下方）：1–2 个 Tag（如「路亚·远投」「新手向」「评测」「开箱」），超出用“+N”收起。

右侧：互动数据紧凑图标（例如：❤️ 点赞数 / 💬 评论数，选择 1–2 个关键指标，避免过多）。

标题展示策略：

标题不再完整展示 2–3 行，避免信息喧宾夺主。

方案 A：标题单行展示在头像和昵称上方或下方，最多 12–16 字，超出省略。

方案 B：将标题弱化为一条短描述文案（从帖子标题中截取），放在底部信息区顶部。

特殊情况处理：

无图帖子：

瀑布流模式中不建议强行展示无图帖子。

可选策略：

a) 使用统一的默认渐变背景 + 帖子标题大字 + 小图标。

b) 将无图帖子仅保留在默认列表模式中，在瀑布流中靠后展示或不展示（视你内容策略）。

多图帖子：

仅使用首图作为背景。

不再展示“3 张图”的布局，以保证瀑布风格统一。

5. 瀑布流交互与滚动体验
5.1 滚动规则

保持筛选条件：

布局切换不会重置筛选条件，如“最新/最热”、“品类筛选”、“标签过滤”等。

位置保持策略：

切换布局时，尽量保持用户当前浏览位置的「内容区间」一致。

实现方式：

记录当前滚动位置与对应的帖子索引。

切换布局后滚动至该索引附近位置，而不是重新回到顶部。

无限滚动加载：

瀑布流模式必须支持下拉触底自动加载下一页。

新加载数据插入两列中时，按「当前累计高度较低的一列优先」的逻辑分配，保证视觉平衡。

5.2 动效优化

卡片加载时使用轻微淡入上移 5–10rpx 的动画即可，避免复杂动画造成性能问题。

新数据加载建议使用 skeleton 骨架屏或简单灰块占位，提升感知流畅度。

6. 性能与实现优化要求
6.1 图片与懒加载

懒加载：

两种布局均开启官方 lazy-load。

优先加载当前视口 + 下方一屏范围内图片。

图片规格约束：

服务端返回适配尺寸的缩略图（宽度按列宽算出，避免在客户端缩放大图）。

控制单张图大小，建议 < 300KB，优先使用 WebP。

预防抖动：

为瀑布流卡片预设高度比例（如固定宽度，按 4:5 / 3:4 比例估算），等图片加载完再微调，减少整体跳动。

或在接口中返回图片宽高比，提前计算卡片高度。

6.2 DOM 与渲染优化

节点数量控制：

瀑布流模式下当列表过长时需考虑“虚拟列表”（如分页渲染、分段插入）或在滚出视口较远时减少不必要节点更新。

避免频繁 setData 全量刷新，仅更新新增数据和必要字段。

布局计算：

推荐使用简单列高记录算法（如 leftColumnHeight / rightColumnHeight），在前端完成卡片列归属分配。

避免每条都用复杂 selectorQuery 多次测量 DOM；尽量基于比例计算 + 少量抽查矫正。

缓存策略：

切换布局时，已经获取的数据不应重复请求。

使用内存缓存分别记录：

当前筛选条件下的默认列表数据

当前筛选条件下的瀑布流列分配结果

减少切换往返造成的多余网络请求和闪白。

7. 可用性与细节要求

可读性：

渐变遮罩必须保证文字与背景对比度充足（推荐对比度 ≥ 4.5:1）。

尽量避免在高亮复杂背景图上直接堆文字。

点击范围：

整个瀑布流卡片区域均可点击进入帖子详情，提升可点区域。

不强制要求局部按钮（如点赞）在列表页操作，如果有，注意不要和卡片跳转冲突。

一致性：

两种布局的核心信息保持一致（同一帖子不会在不同布局显示完全矛盾的数据），只是信息量的浓缩与展现方式不同。

交互语义统一：点击卡片 → 进入详情；下拉刷新 → 当前布局、当前筛选重新拉取。

异常状态处理：

空数据：瀑布流模式下显示简洁插画+提示文案“暂无符合条件的帖子”，附带“去看看最新帖子”按钮。

加载失败：保留重试入口，信息不遮挡已加载内容。

8. 需求优先级建议

如果你要排开发顺序，可以直接按这个顺着做：

P0

布局切换按钮实现（含状态记忆）

瀑布流两列布局 + 一张图背景 + 底部信息区基础样式

保持筛选条件和数据源一致

无限滚动加载 + 懒加载

P1

切换布局时的位置保持

骨架屏/过渡动画

无图帖子适配方案

P2

虚拟列表/性能进一步优化

更精细的列平衡逻辑和预估高度算法

更多个性化样式（官方认证、置顶标识、活动帖特殊样式等）
